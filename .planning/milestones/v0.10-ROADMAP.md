## Phase Details

### Phase 6: Schema, Storage & Safe Foundation

**Goal**: La capa de datos del landing existe y es segura — `landing_config` se almacena, se sirve por la vista acotada sin filtrar secretos, cae a un default seguro ante config malo/null, y las imágenes tienen un bucket público-lectura / privado-escritura namespaced por `business_id`. La garantía de cero-regresión queda probada acá, antes de que exista UI nueva.
**Depends on**: Nothing (first phase of v0.10; builds on v0.9 `public_businesses` view)
**Requirements**: CFG-01, CFG-02, CFG-03, CFG-04, LAND-06
**Success Criteria** (what must be TRUE):

  1. Un `SELECT` con anon-key sobre la tabla base `businesses` es denegado, y la vista `public_businesses` (con `landing_config` agregado por columnas explícitas) NUNCA devuelve `mp_access_token`, `resend_api_key` ni `google_refresh_token` — verificado por el test de aislamiento que extiende TEST-01.
  2. Un `landing_config` malformado o inválido NO tira 500 en `/[slug]` ni rompe el flujo de reservas — `parseLandingConfig()` (Zod `safeParse`) cae a `DEFAULT_LANDING_CONFIG`.
  3. Un negocio con `landing_config` null ve la misma página pública que antes de v0.10, byte-equivalente — cero regresión.
  4. Una imagen subida al bucket `landing-assets` es legible públicamente por URL, pero un intento de escritura fuera del prefijo `{business_id}/` del dueño es rechazado.

**Plans**: 2/2 plans complete

- [x] 06-01-PLAN.md — `lib/landing/schema.ts` (Zod envelope + DEFAULT + total `parseLandingConfig`), `/[slug]` seam (render unchanged), parser unit tests (CFG-01, CFG-03, LAND-06)
- [x] 06-02-PLAN.md — migración 030 (columna + vista acotada + bucket `landing-assets` + RLS owner-write), apply manual [BLOCKING], extensión del test de aislamiento anon-key (CFG-01, CFG-02, CFG-04)

### Phase 7: Renderer & Section Components

**Goal**: El visitante ve un landing real compuesto por las secciones habilitadas del negocio en el orden configurado, con las reservas funcionando integradas como una sección — sin regresión de seña/disponibilidad/MercadoPago. El contenido derivado (Services, Hours, Location) no se duplica, las secciones vacías se ocultan con elegancia, y todo es mobile-first y WCAG AA.
**Depends on**: Phase 6
**Requirements**: LAND-01, LAND-02, LAND-03, LAND-04, LAND-05, LAND-07
**Success Criteria** (what must be TRUE):

  1. El visitante ve las 7 secciones fijas (Hero, About, Services, Gallery, Location, Hours, CTA) renderizadas según `order`/`enabled`, más la sección de reservas.
  2. El visitante puede completar una reserva (con seña vía MercadoPago y sin seña) desde el landing, sin diferencia de comportamiento frente al flujo pre-v0.10 — `BookingClient` queda intacto, contrato de props congelado.
  3. La sección Services se deriva automáticamente de los servicios del negocio, y Hours/Location de la data existente, sin contenido duplicado.
  4. Una sección sin contenido (ej. Gallery sin imágenes) se oculta en vez de mostrar un bloque vacío.
  5. El landing es usable a 375px y pasa WCAG AA (contraste, foco visible, jerarquía de headings correcta).

**Plans**: 4/4 plans complete

- [x] 07-01-PLAN.md — Fundacion testeable: tipos `data` por seccion en `lib/landing/schema.ts` (envelope F6 intacto) + `lib/landing/derive.ts` (orderedSections/groupHoursByDay/predicados empty-state) + tests Vitest + `next.config.ts` images.remotePatterns (Opcion A) (LAND-01, LAND-04, LAND-05)
- [x] 07-02-PLAN.md — Secciones derivadas RSC: Hero (unico h1, nunca oculto), Services (de la tabla), Hours (de time_blocks), Location (itera locations) (LAND-03, LAND-04, LAND-05, LAND-07)

### Phase 8: Theming

**Goal**: Cada negocio puede aplicar uno de 2-3 presets de tema (color, tipografía, radios, espaciado) y overridear tokens puntuales dentro del set permitido, resuelto server-side sin flash de contenido sin estilar, coherente en claro y oscuro — sin que el theming rompa nunca la UI de reservas.
**Depends on**: Phase 7
**Requirements**: THEME-01, THEME-02, THEME-03, THEME-04
**Success Criteria** (what must be TRUE):

  1. Un landing con un preset elegido renderiza con su set de tokens (color/tipografía/radios/espaciado) sin FOUC, vía resolución server-side por el `PaletteScript` existente.
  2. Un override puntual del config (ej. color primario) se aplica sobre el preset sin romper el resto de los tokens.
  3. El tema se ve coherente en modo claro y oscuro.
  4. Dentro del wrapper temado, el date picker (`react-day-picker`), el drawer (`vaul`) y los toasts (`sonner`) siguen funcionales y visibles — el theming no clipea ni rompe la UI de reservas.

**Plans**: 2/2 plans complete

### Phase 9: SEO, OG & Performance

**Goal**: Cada landing sirve metadata por-negocio (title/description), Open Graph con `og:image` del hero (1200x630), y JSON-LD `LocalBusiness`, con el hero cargado optimizado para buen LCP/CLS. Toda la generación de metadata es fail-safe: un config malo produce metadata default válida, nunca un error.
**Depends on**: Phase 8
**Requirements**: SEO-01, SEO-02, SEO-03, SEO-04, SEO-05
**Success Criteria** (what must be TRUE):

  1. Cada landing emite meta title y description derivados de su config, distintos por negocio.
  2. Cada landing emite Open Graph (`og:title`, `og:description`, `og:url`) y `og:image` 1200x630 tomada del hero.
  3. Cada landing emite JSON-LD `LocalBusiness` (nombre, dirección, horarios, teléfono) vía `JSON.stringify` de un objeto tipado.
  4. La imagen del hero carga con dimensiones explícitas (sin CLS) y `priority` para buen LCP.
  5. Un `landing_config` malo produce metadata default válida — `generateMetadata` nunca tira un error.

**Plans**: 3/3 plans complete

### Phase 10: Skill `forjo-web-builder`

**Goal**: El operador corre la skill con el slug de un negocio y obtiene un `landing_config` poblado, con imágenes re-hosteadas en `landing-assets`, validado con el Zod compartido y escrito vía service-role (sin endpoint web de escritura), devolviendo la URL de preview viva. El scrape de Instagram es opcional y degrada con elegancia a input manual.
**Depends on**: Phase 9 (escribe la forma final del config que definen las fases 6-9)
**Requirements**: SKILL-01, SKILL-02, SKILL-03, SKILL-04
**Success Criteria** (what must be TRUE):

  1. Correr la skill con un slug produce una fila `businesses.landing_config` poblada y devuelve la URL de preview viva en `/[slug]`.
  2. La skill scrapea opcionalmente el Instagram del cliente para copy/imágenes y, si el scraping falla, completa igual degradando a input manual.
  3. Las imágenes quedan re-hosteadas en `landing-assets/{businessId}/{uuid}.{ext}` y referenciadas por URL — nunca se hot-linkea el CDN de Instagram en runtime.
  4. La skill valida el config con el esquema Zod compartido y lo escribe vía service-role — no se expone ningún endpoint de escritura a la web.

**Plans**: 3/3 plans complete
