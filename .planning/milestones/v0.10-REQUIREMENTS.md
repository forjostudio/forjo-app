# Requirements: Forjo App — Web Builder (v0.10)

**Defined:** 2026-06-17
**Core Value:** Un negocio NUNCA puede leer ni modificar datos de otro, y los flujos de pago no pueden ser forzados ni falsificados. (v0.10 suma la web de marca por negocio SIN debilitar este invariante.)

## v1 Requirements

Requirements del milestone v0.10. Cada uno mapea a una fase del roadmap. Actores: **dueño** (negocio cuyo landing se configura), **visitante** (usuario anónimo de `/[slug]`), **operador** (Forjo Studio corriendo la skill).

### Config & Storage (CFG)

- [x] **CFG-01**: El landing de un negocio se almacena como un único `landing_config` (jsonb) en su fila, modificable sin deploy de código.
- [x] **CFG-02**: La página pública `/[slug]` lee `landing_config` a través de la vista acotada `public_businesses`, que NUNCA expone columnas secretas al rol `anon`.
- [x] **CFG-03**: Un `landing_config` inválido o malformado cae a un default seguro (Zod `safeParse`) en vez de romper — `/[slug]` y las reservas nunca tiran 500.
- [x] **CFG-04**: Las imágenes del landing se guardan en un bucket de Storage de lectura pública, namespaced por `business_id`, escribible solo por el dueño/service-role.

### Landing Renderer & Sections (LAND)

- [x] **LAND-01**: El visitante ve el landing compuesto por las secciones habilitadas del negocio, en el orden configurado (`order`/`enabled`).
- [x] **LAND-02**: El visitante puede reservar un turno directo desde el landing vía el flujo de reservas existente, integrado como una sección — sin regresión en seña/disponibilidad/MercadoPago.
- [x] **LAND-03**: El renderer soporta las secciones fijas Hero, About, Services, Gallery, Location, Hours y CTA (WhatsApp).
- [x] **LAND-04**: La sección Services se deriva automáticamente de los servicios del negocio; Hours y Location se derivan de la data existente — sin duplicar contenido.
- [x] **LAND-05**: Las secciones sin contenido se ocultan con elegancia (empty-state) en vez de mostrar bloques vacíos.
- [x] **LAND-06**: Un negocio sin `landing_config` (null) ve la misma página pública que antes de v0.10 — cero regresión.
- [x] **LAND-07**: El landing es mobile-first (375px) y cumple WCAG AA (contraste, foco visible, jerarquía de headings).

### Theming (THEME)

- [x] **THEME-01**: El landing de un negocio puede usar uno de 2-3 presets de tema (color, tipografía, radios, espaciado).
- [x] **THEME-02**: Un landing puede overridear tokens puntuales del tema (ej. color primario, tipografías dentro del set permitido) sobre su preset.
- [x] **THEME-03**: El tema seleccionado se aplica server-side sin flash de contenido sin estilar / con tema incorrecto (sin FOUC), coherente en claro y oscuro.
- [x] **THEME-04**: El theming nunca rompe la UI de reservas (date picker, drawer, toasts siguen funcionales dentro del wrapper temado).

### SEO & OG (SEO)

- [x] **SEO-01**: Cada landing sirve meta title y description por-negocio derivados de su config.
- [x] **SEO-02**: Cada landing sirve Open Graph (`og:title`, `og:description`, `og:url`) y `og:image` (1200x630) tomada del hero.
- [x] **SEO-03**: Cada landing emite JSON-LD `LocalBusiness` (nombre, dirección, horarios, teléfono).
- [x] **SEO-04**: La imagen del hero carga optimizada (dimensiones explícitas, sin CLS) para buen LCP.
- [x] **SEO-05**: La generación de metadata es fail-safe — un config malo produce metadata default válida, nunca un error.

### Skill `forjo-web-builder` (SKILL)

- [x] **SKILL-01**: Correr la skill con el slug de un negocio produce un `landing_config` poblado y devuelve la URL de preview viva.
- [x] **SKILL-02**: La skill puede opcionalmente scrapear el Instagram del cliente para copy/imágenes, degradando con elegancia a input manual si el scraping falla.
- [x] **SKILL-03**: La skill re-hostea las imágenes en el bucket `landing-assets` y las referencia por URL — nunca hot-linkea el CDN de Instagram en runtime.
- [x] **SKILL-04**: La skill valida el config con el esquema Zod compartido y lo escribe vía service-role (sin endpoint de escritura expuesto a la web).

## v2 Requirements

Reconocidos pero diferidos — no entran en el roadmap de v0.10.

### Editor en el dashboard (EDIT)

- **EDIT-01**: El dueño puede editar su landing (textos, imágenes, on/off de secciones, tema) desde el dashboard.

### Track operativo (OPS)

- **OPS-01**: Sección de upsell "Tu página web" en el dashboard + `landing_status` (none → requested → paid → live) + aviso a Forjo.
- **OPS-02**: Base URL parametrizado por negocio (reemplaza `NEXT_PUBLIC_APP_URL` global en emails y `back_url` de MercadoPago).
- **OPS-03**: Dominio propio del cliente vía resolución por header `Host` en `proxy.ts` (requiere Vercel Pro).

## Out of Scope

| Feature | Reason |
|---------|--------|
| Editor de landing en el dashboard | Deferido: la skill escribe el config; el editor se suma cuando haya demanda real (v2 EDIT-01) |
| Venta in-app + `landing_status` | Track operativo, no bloquea la capa de plantilla (v2 OPS-01) |
| URLs por-negocio / dominio propio | Track operativo + requiere Vercel Pro (v2 OPS-02/03) |
| Sitio standalone vía API (bespoke total, D6) | Milestone aparte; requiere CORS + widget embebible |
| Layout libre / drag-and-drop por cliente | Anti-feature para esta etapa: el set de secciones es FIJO (D3) |
| Inyectar HTML suelto de `instagram-a-web`/`web-scrolling` | Esas skills son referencia + scraping, no output literal (D5); todo es React/Tailwind nativo |
| Testimonios / reviews | No hay modelo de datos para soportarlo |
| Animaciones de scroll premium en el MVP | P2 diferenciador — se suma post-lanzamiento, detrás de `prefers-reduced-motion`, tras verificar cero regresión de CLS |
