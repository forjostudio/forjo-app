# Roadmap: Forjo App — Security Hardening (v0.9)

## Overview

Milestone de endurecimiento pre-lanzamiento (brownfield). No agrega features: cierra cinco agujeros de seguridad auditados antes de salir a producción con clientes reales. El recorrido va de la raíz hacia afuera: primero se sella el aislamiento de datos y secretos en la capa de DB (Fase 1), luego se asegura la integridad de los pagos que dependen de ese cambio (Fase 2), después se endurecen las superficies de ataque independientes —endpoints admin (Fase 3) y gating de plan en booking público (Fase 4)— y finalmente se introduce una suite de tests que congela el comportamiento correcto que dejaron las fases anteriores (Fase 5). El invariante que sostiene todo: un negocio nunca puede leer ni modificar datos de otro, y los flujos de pago no pueden ser forzados ni falsificados.

## Phase Details

### Phase 1: RLS Lockdown + Secret Isolation

**Goal**: El rol `anon` no puede leer columnas sensibles de `businesses`, `services` ni `business_hours`; los secretos viven en `business_secrets` con RLS solo-dueño, y la página pública sigue funcionando leyendo solo vistas acotadas
**Mode:** mvp
**Depends on**: Nothing (first phase)
**Requirements**: SEC-01
**Success Criteria** (what must be TRUE):

  1. Un cliente con `anon` key no puede hacer `SELECT` de columnas sensibles directamente sobre `businesses`, `services` ni `business_hours` (la policy `USING(true)` ya no existe en ninguna de las tres)
  2. Los cinco secretos (`mp_access_token`, `mp_refresh_token`, `resend_api_key`, `recaptcha_secret_key`, `google_refresh_token`) residen en `business_secrets`, legible solo por el dueño del negocio, no por `anon`
  3. La página pública de reservas `/[slug]` sigue cargando datos de negocio, servicios y horarios leyendo desde las vistas acotadas (no desde las tablas base)
  4. Todos los lectores de secretos (deposit webhook, `lib/payment.ts`, `lib/recaptcha.ts`, `lib/email.ts`, `lib/google-calendar.ts`) leen de `business_secrets` y la app no rompe tras el deploy coordinado migración + código
  5. Se resolvió explícitamente (sí/no, con criterio documentado) si corresponde rotar claves reales por exposición previa en la URL pública de Vercel

### Phase 2: MP Webhook Signature + Amount Check

**Goal**: El webhook de seña rechaza firmas inválidas (fail-closed) y no confirma turnos cuando el monto pagado no coincide con la seña esperada; ambos webhooks comparten un único verificador de firma
**Mode:** mvp
**Depends on**: Phase 1 (el deposit webhook lee secretos desde `business_secrets`, que solo existe tras el move de Fase 1; ambas fases comparten superficie de deploy atómica)
**Requirements**: SEC-02
**Success Criteria** (what must be TRUE):

  1. Una notificación al webhook de seña con firma `x-signature` ausente o inválida es rechazada (fail-closed), sin confirmar el turno
  2. Un pago aprobado por un monto distinto a la seña esperada (`transaction_amount` ≠ `deposit_amount`, comparado en enteros-centavos) NO confirma el turno
  3. La firma se valida con el `data.id` tomado del query string (lowercased, manifest correcto), igual que el webhook de suscripción ya validado
  4. `verifyMPSignature` vive en `lib/mercadopago.ts` y es el único verificador importado por ambos webhooks (suscripción y seña), sin duplicación inline

### Phase 3: Admin Endpoint Hardening

**Goal**: Los endpoints admin (`set-plan`, `setup-plans`) solo aceptan el secreto por header HTTP con comparación en tiempo constante; el secreto nunca queda en logs de acceso
**Mode:** mvp
**Depends on**: Phase 1 (puede ejecutarse en cualquier hueco; sin dependencia real de archivos sobre Fases 1/2/4)
**Requirements**: SEC-03
**Success Criteria** (what must be TRUE):

  1. `setup-plans` ya no acepta el secreto por query string (`?secret=`); solo lo lee del header HTTP
  2. Ambos endpoints admin comparan el secreto con `crypto.timingSafeEqual` usando el patrón hash-both-sides (SHA-256 a ambos lados)
  3. Un header de secreto basura o de longitud distinta devuelve 401, nunca un 500 por `ERR_CRYPTO_TIMING_SAFE_EQUAL_LENGTH`
  4. El secreto admin no aparece en los logs de acceso de Vercel para ninguna de las dos rutas

**Plans**: 1/1 plans complete

### Phase 4: Plan Gating on Public Booking

**Goal**: El endpoint público `/api/booking/create` rechaza reservas (403) si el negocio tiene `plan_status` en `expired` o `cancelled`, sin afectar a `trial` ni `active`
**Mode:** mvp
**Depends on**: Phase 1 (`plan_status` sigue disponible en el read server-side tras el move; puede ejecutarse en paralelo con Fase 3)
**Requirements**: SEC-04
**Success Criteria** (what must be TRUE):

  1. Una reserva pública contra un negocio con `plan_status` `expired` o `cancelled` recibe 403 con código `plan_inactive`
  2. Una reserva contra un negocio con `plan_status` `trial` o `active` se procesa normalmente (sin sobre-bloqueo por blocklist explícito `['expired', 'cancelled']`)
  3. El gating no agrega un round-trip adicional: reaprovecha el lookup de negocio por slug ya existente

**Plans**: 1/1 plans complete

### Phase 5: Vitest Test Suite

**Goal**: Existe una suite de tests Vitest ejecutable en CI que prueba el aislamiento multi-tenant (lectura/escritura cruzada con cliente anon-key) y los dos webhooks de pago; los tests se ponen rojos si una policy de aislamiento se cae o un webhook deja de rechazar forjas
**Mode:** mvp
**Depends on**: Phases 1-4 (los tests asertan el comportamiento corregido que solo existe tras cerrar los agujeros de las fases anteriores)
**Requirements**: TEST-01
**Success Criteria** (what must be TRUE):

  1. `vitest run` ejecuta la suite completa en CI con `environment: node`, sin llamadas a APIs en vivo (`mpFetch` mockeado)
  2. Los tests de aislamiento confirman que una lectura cruzada entre dos negocios devuelve 0 filas, ejecutados con cliente anon-key y SIN filtro `business_id` en la query de aserción (un guard falla ruidosamente si el cliente es service-role)
  3. Una escritura cruzada entre tenants (negocio A intentando modificar datos de negocio B con anon-key) es rechazada por RLS
  4. Ambos webhooks de MP tienen cobertura: firma válida pasa, firma ausente/inválida devuelve 401, y un mismatch de monto en el webhook de seña deja el turno SIN confirmar
