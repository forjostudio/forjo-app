# Requirements: Forjo App — Security Hardening (v0.9)

**Defined:** 2026-06-15
**Core Value:** Un negocio NUNCA puede leer ni modificar datos de otro, y los flujos de pago no pueden ser forzados ni falsificados.

## v1 Requirements

Requisitos para el milestone de endurecimiento pre-lanzamiento. Los cinco son bloqueantes del lanzamiento; ninguno se difiere.

### Aislamiento multi-tenant (RLS)

- [x] **SEC-01**: El rol `anon` no puede leer columnas sensibles de `businesses`, `services` ni `business_hours`; los datos públicos se exponen solo vía vistas acotadas, y los secretos (`mp_access_token`, `mp_refresh_token`, `resend_api_key`, `recaptcha_secret_key`, `google_refresh_token`) viven en una tabla `business_secrets` con RLS solo-dueño

### Pagos (webhooks MercadoPago)

- [x] **SEC-02**: Ambos webhooks de MercadoPago validan la firma `x-signature` fail-closed; el webhook de seña compara el monto pagado (`transaction_amount`) contra la seña esperada (`deposit_amount`) antes de confirmar el turno

### Endpoints admin

- [x] **SEC-03**: Los endpoints admin (`set-plan`, `setup-plans`) se autentican solo por header HTTP con comparación en tiempo constante (`crypto.timingSafeEqual`, hash-both-sides), sin aceptar el secreto por query string

### Booking público

- [x] **SEC-04**: El endpoint público `/api/booking/create` rechaza reservas (403) si el negocio tiene `plan_status` en `expired` o `cancelled`; `trial` y `active` no se ven afectados

### Testing

- [x] **TEST-01**: Existe una suite de tests automatizada (Vitest), ejecutable en CI, que cubre el aislamiento multi-tenant (lectura/escritura cruzada entre dos negocios con cliente anon-key) y los dos webhooks de pago (firma válida/inválida + mismatch de monto)

## v2 Requirements

Ninguno. Este milestone no defiere requisitos a v2 — los cinco son pre-requisito de lanzamiento. Los ítems de robustez/deuda técnica están en Out of Scope.

## Out of Scope

Robustez o deuda técnica, no agujeros de seguridad. No bloquean el lanzamiento (backlog v2).

| Feature | Reason |
|---------|--------|
| Tabla `webhook_events` para idempotencia fuerte de webhooks | Robustez, no agujero de seguridad; la firma + monto ya cierran el riesgo de forja |
| Rate-limiting por IP en `/api/booking/create` | Mitigación de abuso, no aislamiento entre tenants |
| Unificar las tres fuentes de verdad de planes (`lib/plans.ts`, `lib/subscription-plans.ts`, `lib/plan-limits.ts`) | Deuda técnica; no afecta seguridad |
| Adoptar Supabase CLI migrations o tabla de versionado de esquema | Tooling de DB; no bloquea el hardening |
| Tipar `mpFetch` / adoptar SDK oficial de MercadoPago | Type-safety; el patrón por `fetch` ya funciona |
| Centralizar la lógica de expiración de turnos (replicada en 3 lugares) | Refactor; fuera del alcance de seguridad |
| Tests de UI / E2E (Testing Library, Playwright) | TEST-01 cubre aislamiento y webhooks con `environment: node`; UI/E2E se defiere |
