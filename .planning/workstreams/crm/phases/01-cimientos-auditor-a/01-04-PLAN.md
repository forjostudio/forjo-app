## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server action (vía ConfirmDialog) | La confirmación es UI; un request directo puede saltarla — la autorización real debe estar server-side |
| operador local (CLI) → app_metadata.is_admin | Otorgar is_admin es la elevación de privilegio más sensible; debe ocurrir fuera del runtime web |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Elevation of Privilege | ConfirmDialog como falsa garantía | mitigate | El dialog NUNCA autoriza; cada acción peligrosa re-valida con `requireAdmin()` server-side (plan 01-01, Pitfall 2). La UI deshabilitada es refuerzo. |
| T-01-12 | Tampering | doble submit de acción peligrosa | mitigate | Estado `loading` deshabilita el botón y bloquea el cierre del dialog → no se dispara onConfirm dos veces. |
| T-01-13 | Elevation of Privilege | otorgar is_admin (llaves del reino) | mitigate | Bootstrap solo vía script local con service-role, fuera del runtime web; no self-serve ni endpoint (§7, D2). Cuando exista la acción de otorgar is_admin en el panel, se auditará (FND-02) — fuera de scope de Phase 1. |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | Phase 1 no instala paquetes nuevos (RESEARCH §"Package Legitimacy Audit": no aplica); `tsx` y `@supabase/supabase-js` ya están. |
