<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente autenticado → DB (audit_log) | Un usuario logueado podría intentar leer o falsificar el log de auditoría vía la anon key / RLS |
| server action/route handler → DB (service-role) | El service-role bypassa RLS; debe usarse solo server-side para escribir audit_log |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | RLS SELECT de audit_log | mitigate | Policy SELECT restringida a `is_admin` vía app_metadata JWT; NO `using(true)` (Pitfall 4, lección migración 029). Un no-admin ve 0 filas. |
| T-01-02 | Tampering / Repudiation | escritura de audit_log | mitigate | Sin policy de insert/update/delete para authenticated/anon → solo `createAdminClient()` (service-role, server-only) escribe vía `logAudit`. El audit no es falsificable por un cliente. |
| T-01-03 | Elevation of Privilege | requireAdmin() en actions | mitigate | `requireAdmin()` re-valida is_admin server-side y LANZA forbidden si falta; se invoca al inicio de cada action/route handler del CRM (D4, Pitfall 2). La UI es solo refuerzo. |
| T-01-04 | Spoofing | fuente de is_admin | mitigate | is_admin se lee de `auth.users.app_metadata` (no editable por el dueño, solo service-role/admin API — D1), no de una columna que el tenant pueda tocar vía RLS. |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | Phase 1 no instala paquetes nuevos (RESEARCH §"Package Legitimacy Audit": no aplica). Sin superficie de supply-chain en este plan. |
</threat_model>
