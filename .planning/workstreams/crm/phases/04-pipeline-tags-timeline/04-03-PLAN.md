<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operador → crm_timeline VIEW | La lectura debe pasar por la RLS admin-read; la VIEW es security_invoker y se lee con session client |
| cliente → content actions | createNote/deleteNote/createTask/completeTask son endpoints POST invocables directo |
| id de la URL → lectura del timeline | El id viene del cliente; cada lectura debe filtrar por business_id (sin fuga cross-entidad) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Information Disclosure | lectura de crm_timeline | mitigate | Se lee con SESSION client (createClient) → la RLS admin-read se hereda via security_invoker; leer con service-role bypassaria el gate (high). Filtro .eq('business_id', id) evita fuga cross-entidad |
| T-04-11 | Tampering | content actions invocadas directo | mitigate | requireAdmin() primera linea + zod.parse segunda en createNote/editNote/deleteNote/createTask/completeTask |
| T-04-12 | Repudiation | notas/tareas creadas/borradas sin traza | mitigate | logAudit en cada content action (note.create/edit/delete, task.create/complete) → alimenta el propio timeline |
| T-04-13 | Information Disclosure | tags de otra entidad en el directorio | mitigate | entity_tags admin-only por RLS; la lectura por entity_type='business' va con service-role tras el guard del layout; tagIds que cruzan al cliente no son sensibles |
| T-04-14 | Tampering | borrar nota (destructivo) | mitigate | deleteNote risk medio detras de ConfirmDialog (refuerzo) + requireAdmin (garantia) |
</threat_model>
