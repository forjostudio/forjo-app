Cierre de los 3 gaps abiertos del re-test del UAT de la Phase 4 (el usuario eligió arreglarlos juntos en UN plan):

- **Gap test 7 (MAJOR) — no hay UI para asignar tags a deals/leads en el pipeline.** Los chips por tarjeta + el filtro OR ya existen (`pipeline-client.tsx`), pero ninguna tarjeta puede recibir tags: el pipeline lee `entity_tags` con `entity_type='lead'` (`pipeline/page.tsx:86,106`) mientras la única UI de asignación es la ficha con `entity_type='business'`. `assignTag`/`removeTag`/`createTag` (`_tag-actions.ts`) ya son genéricos (aceptan lead|business) — falta el afordance en el tablero. Decisión de diagnóstico: agregar un control de tags por tarjeta que REUSE el diálogo de tags de la ficha, parametrizado por `entityType`, para no duplicar UI.
- **Gap test 13 (minor) — la tag recién creada no se auto-asigna.** `createTag` (`_tag-actions.ts:34`) retorna void; tras crear hay que tocar la tag en "asignar existente". Decisión: `createTag` devuelve el id; el handler encadena `assignTag` con ese id + refresh (asignación en un paso). Aplica tanto a la ficha como al nuevo diálogo del pipeline → vive UNA sola vez en el diálogo compartido.
- **Gap test 14 (minor) — "Limpiar filtros" solo en empty-state.** El botón `clearFilters` solo se renderiza dentro del EmptyState (`negocios-client.tsx:288-296`). Decisión: mostrar un control "Limpiar filtros" cuando hay filtro activo (`selectedTagIds.length>0` o query/tab != default), independiente del empty-state.

Estrategia clave (anti-duplicación, alineada con la diagnosis): **extraer el diálogo de tags de la ficha a un componente compartido `components/crm/tag-manager-dialog.tsx` parametrizado por `entityType` ('lead' | 'business')**, con la auto-asignación de la tag recién creada (gap 13) implementada UNA vez adentro. La ficha pasa a consumir ese componente (`entityType='business'`); el pipeline lo consume por tarjeta (`entityType='lead'`, `entityId=deal.leadId`). Así el gap 7 y el gap 13 se cierran con una sola pieza de UI reusada por ambos surfaces.

Purpose: el feature de tags del pipeline queda funcional end-to-end (asignar → filtrar), la creación de tags es de un solo paso en ambos surfaces, y el directorio tiene un afordance de limpiar filtros siempre visible.
Output: schema/action de tags (createTag→id), diálogo compartido nuevo, ficha refactorizada a consumirlo, pipeline (page + client) con afordance de tags por tarjeta, directorio con clear-filters.
