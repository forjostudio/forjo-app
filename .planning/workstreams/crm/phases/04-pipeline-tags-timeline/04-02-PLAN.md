<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| onboarding (sesion owner) → tablas admin-only | El flujo de signup corre bajo la sesion anon del dueno; escribir leads/deals exige cruzar a service-role server-side |
| cliente del tablero → server action | moveStage/markLost/convertLead son endpoints POST invocables directo, sin pasar por la UI |
| input del cliente → conversion | El email/lead_id no son confiables; deben re-derivarse server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Elevation of Privilege | linkLeadOnSignup (escritura desde flujo no-admin) | mitigate | El action corre service-role server-side; el email del owner se re-deriva de supabase.auth.getUser() (NUNCA del input); es la unica via donde un flujo no-admin toca tablas admin-only, minima y aislada (high) |
| T-04-06 | Spoofing | conversion (email/lead_id del cliente) | mitigate | linkLeadOnSignup ignora cualquier email/leadId del input y matchea por el email autenticado del owner (regla anti-tampering del booking) |
| T-04-07 | Tampering | moveStage/markLost/convertLead invocadas directo | mitigate | requireAdmin() primera linea de cada action + zod.parse segunda; el ConfirmDialog/UI es solo refuerzo |
| T-04-08 | Repudiation | cambios de etapa / conversion / lost sin traza | mitigate | logAudit en cada mutacion con action code exacto + metadata {from,to}/{reason}; alimenta el timeline (Plan 03) |
| T-04-09 | Denial of Service | conversion rompe el onboarding | accept/mitigate | La llamada es best-effort en try/catch: un fallo no bloquea la creacion del negocio ni el redirect (riesgo aceptado: un lead puede quedar sin vincular y se re-vincula a mano, D-06) |
</threat_model>
