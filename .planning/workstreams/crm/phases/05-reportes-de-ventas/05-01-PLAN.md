<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Vercel cron → API route (`/api/cron/cancel-expired`) | Invocación externa; entra con `Authorization: Bearer ${CRON_SECRET}` |
| API route (service-role) → Postgres | El cron escribe `mrr_snapshots` con service-role (no hay sesión admin en el cron) |
| Postgres RLS (`mrr_snapshots`) → cualquier lectura de sesión | La policy admin-read `is_admin` es el gate; sin policy de write nadie salvo service-role escribe |
| Operador (`is_admin`) → tablas admin-only | El aislamiento es el flag `is_admin` del JWT, NO `business_id` (datos cross-tenant del operador por diseño) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Elevation of Privilege | `mrr_snapshots` RLS | mitigate | Una sola policy `for select using ((select auth.jwt() -> 'app_metadata' ->> 'is_admin') = 'true')` (verbatim de 034); SIN policy de insert/update/delete. NUNCA `using(true)` (lección 029). Verificado por grep en Task 2. |
| T-05-02 | Tampering | snapshot mensual | mitigate | Sin policy de insert/update/delete en `mrr_snapshots` → solo el service-role del cron escribe; el dueño de un negocio no puede falsificar ni alterar su MRR histórico. PK `(month,plan)` impide duplicar filas. |
| T-05-03 | Spoofing | cron `/api/cron/cancel-expired` | mitigate | El handler ya valida `Authorization: Bearer ${CRON_SECRET}` (route.ts:8); el bloque de snapshot hereda esa barrera (mismo handler, no se relaja). Sin nuevo endpoint que abra superficie. |
| T-05-04 | Denial of Service | cron piggyback | mitigate/accept | El bloque de snapshot va en su propio try/catch (best-effort, criterio del loop de emails): un fallo del upsert loguea y NO aborta cancel-expired. Residual aceptado: si un día falla, el mes en curso se re-escribe al día siguiente (idempotente por la PK). |
| T-05-SC | Tampering (supply-chain) | npm installs | mitigate | Cero dependencias nuevas (D-09: recharts ^3.8.1 ya instalado; vitest ya instalado). No se toca `package.json`/`package-lock.json` en este plan → sin superficie de cadena de suministro. Sin `[ASSUMED]`/`[SUS]` packages (RESEARCH Package Legitimacy Audit). |

Notas de aislamiento (T-05): `business_id` NO aparece en `mrr_snapshots` ni en sus lecturas como scope de tenant — el gate es `is_admin` (datos del operador, cross-tenant por diseño, D-10).
</threat_model>
