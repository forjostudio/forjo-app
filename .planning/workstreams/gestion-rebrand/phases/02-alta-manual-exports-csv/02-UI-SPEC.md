# Phase 2 — UI Design Contract

> Visual and interaction contract for **alta manual de cliente** (CLIENT-01), the **badge de origen** (SC-2), and the **botones de export CSV** (DATA-01, DATA-02) of Forjo Gestión.
> Generated by gsd-ui-researcher.

**Scope note (read first):** This is a **small UI surface over an already-shipped design system** (Bauhaus theme + shadcn `base-nova`/`neutral`, live in `app/globals.css`). It is **NOT a visual redesign**. This contract governs **LAYOUT · STRUCTURE · INTERACTION** of exactly three surfaces: (1) a dedicated "Nuevo cliente" form/modal on `/clients`, (2) an origin badge per client row, (3) two "Exportar CSV" buttons (Clientes + Finanzas). Colors, fonts, spacing scale, and the 60/30/10 discipline are **inherited** from the Phase 1 UI-SPEC (`../01-reorg-de-ia-ayuda/01-UI-SPEC.md`) — not re-chosen here. Reuse-only: no new components, no new dependencies. UI fidelity + visual consistency with sibling components is load-bearing for this developer.

## Spacing Scale

Inherited (all multiples of 4 — matches existing Tailwind v4 usage in `clients-client.tsx` / `finances-client.tsx`). This phase declares only the values it uses.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (`gap-1`) | Tight item stacking within a group; badge icon-to-label gap |
| sm | 8px (`gap-2`, `space-y-2`, `p-2`) | Icon-to-label gap inside "Exportar CSV" / "Nuevo cliente" buttons; form field vertical rhythm, header action-row gap, dialog button row |
| md | 12px (`gap-3`, `px-3`, `py-3`) | Client row inner padding (existing), dialog header padding |
| md+ | 16px (`p-4`, `space-y-4`, `mt-4`) | Left-panel header padding (existing `p-4`), form section rhythm inside the modal |
| lg+ | 24px (`space-y-6`, `p-6`) | Finances page section rhythm (existing) |

Exceptions:
- **Touch targets ≥ 44px:** the "Nuevo cliente" and "Exportar CSV" buttons must present a ≥ 44px effective hit area on 375px mobile. Reuse `Button` `size="sm"` (`h-8` = 32px visual) only where the existing screen already does; for the **primary alta CTA on mobile** prefer default `size` (`h-9`+) or ensure the row padding brings the hit area to ≥ 44px. Icon-only variants are NOT used for these actions (labels are required — see Copywriting).
- Origin badge reuses `Badge`'s fixed `h-5` (20px) — it is a **status indicator, not a touch target**, so the 44px rule does not apply (it is non-interactive).

| Role | Token | Usage in this phase |

**Accent reserved for:** `Nuevo cliente` CTA · alta modal submit button · the **"Manual"** origin badge only. Never apply `--primary` to the "Reserva" badge, the "Importado" badge, the export buttons, or field labels.

### Origin badge color mapping (SC-2 — decided, not asked)

Decision rationale: CONTEXT D-01 + scope note delegate the badge variant to Claude's discretion, and the design system constrains the answer (accent not wasted, 3 states distinguishable, WCAG AA, reuse-only `Badge`). The three states map to **existing `Badge` variants** — no invented colors, no new CSS:

| Origin value (`clients.origin`) | Label | `Badge` variant | Resolved look | Why |
|--------------------------------|-------|-----------------|---------------|-----|
| `reserva` (default / backfill) | **Reserva** | `outline` | `border-border text-foreground` on canvas | Most common state (came via booking) → the quiet, neutral one; does not compete for attention |
| `manual` (alta escribe esto) | **Manual** | `default` | `bg-primary text-primary-foreground` | The action this phase introduces; the one worth spotting. This is the **only** accent spend here, within the 10% budget |
| `importado` (reservado Fase 3) | **Importado** | `secondary` | `bg-secondary text-secondary-foreground` | Distinct from both above (filled but neutral), lives in the 30% band; ready for Fase 3 with no further design work |

- All three variants are already WCAG-AA in both light/dark palettes (shipped tokens). `outline` vs `secondary` vs `default` differ by **border/fill/color**, not by hue alone → distinguishable without relying on color perception.
- Badge is **non-interactive** (a status label). Add `title={label}`/`aria-label` is optional; the visible text already conveys meaning (no icon-only, no color-only signal).
- **Placement in the list:** the client row in `clients-client.tsx` (the `<button>` at L519–543) currently shows name + `fichaNum · N visitas` + a status **dot** on the right. Add the origin badge in the info block **under the name / next to the ficha meta line** (e.g. inline after `fichaNum(num) · N visitas`, or on its own line in the `flex-1 min-w-0` column). Do **not** replace or restyle the existing status dot (client lifecycle status ≠ origin — they coexist). Keep the row height stable; the badge's `h-5` fits the existing `py-3` row.

**Contrast:** all inherited pairs pass WCAG AA. New surfaces (form fields, dialog) use `text-foreground` on `bg-card`/`bg-background`, `text-muted-foreground` for helper text, `text-destructive` for errors — verify ≥ 4.5:1 (inherited defaults already pass).

## Interaction Contracts

### A. Alta manual de cliente (CLIENT-01)

**Pattern choice — Dialog (modal), NOT an inline form.** Rationale: `clients-client.tsx` is a two-panel master/detail layout with no free vertical space for an inline "new" form, and it **already uses the shadcn `Dialog`** for both "¿Eliminar cliente?" (L786) and "Fusionar duplicados" (L802). Mirror that existing pattern → a third `Dialog`. This is the dominant pattern on the screen; do not introduce a new surface type.

**Trigger:** a **"Nuevo cliente"** button (primary CTA) in the **left-panel header** of `clients-client.tsx` (the `p-4 border-b` block, L417), on the header row that currently holds the `h1` + the merge-duplicates icon (L418–427). Place it as the primary action of that row (e.g. right-aligned, `size="sm"` default `bg-primary`, icon `UserPlus` from lucide). It must reach ≥ 44px hit area on mobile.

| Aspect | Contract |
|--------|----------|
| Open | Click "Nuevo cliente" → opens `Dialog` (`DialogContent className="sm:max-w-md"`, matching the merge dialog). Title `DialogTitle` = **"Nuevo cliente"** (or `term.client` if terminology-driven — use `Nuevo ${term.client}` to respect vertical terminology, e.g. "Nuevo paciente" in salud). |
| Fields (order) | 1. **Nombre** (required) · 2. **Teléfono** · 3. **Email** — with a rule "al menos uno de teléfono o email". 4. **Notas** (optional, `Textarea`). 5. **Obra social** (`insurance_name`) + **N° de afiliado** (`insurance_number`), **rendered ONLY when `isSalud`** (`useVertical().key === 'salud'`) — same gating the inline edit form already uses (L601). One field per line on mobile; teléfono+email MAY sit on one row on desktop (mirrors existing edit form L595–600). |
| Labels | **Always visible** `Label` above each input (never placeholder-only). Mark optional fields with "(opcional)"; do NOT asterisk required ones. |
| Validation | `zod` schema + `zodResolver`; **inline validation onBlur** (not only on submit). Rule set: `name` non-empty; at least one of `phone`/`email` present; `email` must be a valid email if filled. Errors render **inline under the field** in `text-xs text-destructive`, immediate. |
| Submit | Primary `Button` **"Guardar"** (`bg-primary`), full-intent. On click: `setSaving(true)`, disable button + show "Guardando..." label to prevent double-submit (existing pattern, L610 / L794). |
| Write path | Server-side (route handler or server action, Claude's discretion per D-02) that **derives `business_id` from the session** and inserts with `origin='manual'`. Never trust a client-supplied `business_id` (tenant isolation). |
| Success | Close dialog, **prepend/insert the new client into local `clients` state** so it appears in the list instantly (SC-1), `toast.success('Cliente creado')` (or `${term.client} creado/a`). |
| Error | Keep dialog open, re-enable submit, `toast.error(...)` with a specific message (see Copywriting). Do not lose entered data. |
| Cancel | `Button variant="outline"` **"Cancelar"** + `Dialog` `onOpenChange` close (click outside / Escape) — native `Dialog` behavior, do not override. |
| States | Submit button: default · hover · focus-visible · disabled (saving). Inputs: default · focus-visible ring (`focus-visible:ring-[3px] ring-ring/50`, shadcn default) · invalid (`aria-invalid`, `text-destructive` message). |

### B. Origin badge (SC-2)

See **Color → Origin badge color mapping** above (variant per state is the load-bearing decision). Interaction: **none — non-interactive status label.** No hover, no click, no focus. One badge per client in the list. Existing status **dot** is untouched and coexists. The detail panel (right panel) MAY also show the badge near the ficha header (`FICHA #NNN` line, L588) for consistency, but the **list badge (SC-2) is the required one**.

### C. Export CSV buttons (DATA-01, DATA-02)

Two **secondary** actions — they must NOT compete with each screen's primary CTA.

| | Clientes (DATA-01) | Finanzas (DATA-02) |
|--|--------------------|--------------------|
| Placement | Left-panel header row of `clients-client.tsx` (L418), next to "Nuevo cliente" and the merge icon. Since "Nuevo cliente" is the primary here, export is the **secondary** (`variant="outline"`). If the row gets crowded on mobile, export MAY collapse to icon+tooltip, but **desktop keeps the text label**. | Existing header action row (`finances-client.tsx` L603), alongside the period selector + "Personalizado" toggle. `variant="outline"`, `size="sm"` (matches the sibling controls at L606–612). |
| Label | **"Exportar CSV"** (verb + context). | **"Exportar CSV"**. |
| Icon | lucide `Download`, `w-4 h-4`, leading. Icon-to-label spacing is the shadcn `Button` component's own built-in gap (inherited from `components/ui/button.tsx` — do NOT set a manual `gap-*` on the button). | lucide `Download`, same treatment. |
| Trigger | Navigates/downloads via the authenticated export route handler (D-04). Prefer an `<a href="/api/export/clients" download>` styled as the Button, OR a click handler that `fetch`es + triggers the download — Claude's discretion. | Same for `/api/export/finances`. |
| Loading | If a click-handler download is used: `setExporting(true)` → disable button + label **"Exportando..."** (prevents double-download), re-enable on completion/error. If a plain `<a download>` is used (instant), no loading state needed. |
| Empty data | Button stays enabled; server returns a CSV with only the header row (valid empty export). No special empty-state UI required. Optionally `toast` "No hay datos para exportar" if the dataset is empty — Claude's discretion. |
| Error | `toast.error('No se pudo exportar. Intentá de nuevo.')`; re-enable the button. |
| States | default · hover · focus-visible · disabled (exporting). Secondary visual weight throughout (`outline`), never `bg-primary`. |
