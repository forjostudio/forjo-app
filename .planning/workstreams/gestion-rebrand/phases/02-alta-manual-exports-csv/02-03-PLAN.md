<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente (browser) → API export | request de descarga del dueño; el tenant a exportar NO puede venir del querystring |
| API export → DB | la query de export cruza a las tablas de tenant; debe filtrar por business_id de la sesión |
| CSV → Excel | contenido de campos (nombres/notas con comas/comillas/acentos) cruza al archivo — riesgo de corrupción de formato, no de seguridad |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Spoofing | GET /api/export/* sin sesión | mitigate | auth gate `auth.getUser()` → 401 `unauthorized` sin sesión en ambos endpoints |
| T-02-11 | Information Disclosure | exportar filas de otro tenant | mitigate | el `business_id` se re-deriva de `owner_id` de la sesión y toda query filtra `.eq('business_id', business.id)`; SIN business_id del querystring (contribution de security); anon+RLS como red de defensa (skill supabase-multitenant-rls) |
| T-02-12 | Elevation | uso de service-role en los exports | mitigate | prohibido: ambos endpoints usan `@/lib/supabase/server` (anon+RLS); grep de verificación confirma que NO importan `supabase/admin` |
| T-02-13 | Tampering | inyección de fórmula / corrupción de CSV vía campos con coma/comilla/salto | mitigate | escaping RFC4180 (`esc` = campo entre comillas dobles + comilla interna duplicada) en todos los campos; BOM UTF-8 para acentos correctos en Excel-AR |
| T-02-14 | Information Disclosure | datos sensibles (obra social) en el CSV de clientes | accept | el CSV lo descarga el propio dueño de sus propios clientes (dato ya visible en su panel); no cruza a otro tenant; el header de obra social solo lleva datos del negocio del actor |
</threat_model>
