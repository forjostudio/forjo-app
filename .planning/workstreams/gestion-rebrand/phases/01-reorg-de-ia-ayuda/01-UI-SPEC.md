# Phase 1 — UI Design Contract

> Visual and interaction contract for the **behavior-frozen navigation reorg** (NAV-01, NAV-02) + **static FAQ** (HELP-01) of Forjo Gestión.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note (read first):** This phase is a **navigation/structure reorg over an already-shipped design system**, NOT a visual redesign. The Bauhaus visual theme (tokens, fonts, palettes, dark mode) is already live in `app/globals.css` and applied across the 11 screens. This contract therefore governs **LAYOUT · STRUCTURE · INTERACTION** of the grouped sidebar, the Negocio hub tabs, and the FAQ disclosure — **not** colors/fonts (which are inherited, not chosen here). The approved mock (`../gestion-actual/Forjo Gestión - Reestructura (offline).html`) + screen inventory (`../gestion-actual/screenshots/inventario-pantallas.md`) are the source of truth for the "de dónde a dónde" map; reproduce them faithfully (UI fidelity is load-bearing for this developer).

## Spacing Scale

Declared values (all multiples of 4 — matches existing Tailwind v4 usage in `sidebar.tsx` / `settings-client.tsx`):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (`gap-1`, `space-y-1`) | Icon-to-label gaps inside nav rows, item stacking within a group |
| sm | 8px (`gap-2`, `p-2`, `space-y-0.5→2`) | Nav row padding, group header padding, tab trigger gaps |
| md | 12px (`gap-3`, `px-3`, `py-2`) | Nav row inner padding (icon + label), card list rows |
| md+ | 16px (`p-4`, `mt-4`, `space-y-4`) | Tab content top margin, FAQ item vertical rhythm, card gaps |
| lg | 20px (`p-5`) | Sidebar brand header padding (matches existing) |
| lg+ | 24px (`p-6`, `space-y-6`) | Card padding, page section vertical rhythm |
| xl | 32px | Reserved (page-level) — not introduced by this phase |

**Group spacing (grouped sidebar, LOCKED to CRM pattern):** group header = `px-2 pt-4 pb-1`; items within group = `space-y-0.5`; nav row = `px-2 py-2 rounded-lg` (or preserve current `px-3 py-2` — see fidelity note). First group has no top divider; groups separate by the `pt-4` on each group header (no explicit `<hr>`).

Exceptions:
- Touch targets: mobile nav rows and tab triggers must be **≥ 44px tall** effective hit area (achieved via `py-2` + line-height; verify on the 375px drawer). Icon-only mobile buttons (`Menu`/`X`) already use `size="icon"` ≥ 44px — preserve.
- Sidebar fixed width `w-60` (240px) desktop / `w-64` (256px) mobile drawer — **preserve exactly** (both `sidebar.tsx` and `crm-sidebar.tsx` use `w-60` desktop; keep whatever the current dashboard sidebar uses).

## Typography

Inherited from the shipped theme — this phase declares **roles/usage**, not new sizes. Do not introduce sizes outside this set.

| Role | Size | Weight | Line Height | Where |
|------|------|--------|-------------|-------|
| Page title (h1) | 24px (`text-2xl`) | 700 (`font-bold`) | ~1.2 | Negocio / Configuración / Ayuda headers (existing pattern in settings-client L798) |
| Section eyebrow | 11px (`text-[11px]`) | 600, `uppercase tracking-[0.16em]` | 1 | `PageEyebrow` above each h1 |
| Sidebar group label | 11px (`text-[11px]`) | 400, `--font-geist-mono`, `uppercase tracking-wider` | ~1 | PANEL · AGENDA · GESTIÓN · REPORTES · AJUSTES headers (mirror crm-sidebar L175) |
| Nav row / tab trigger | 14px (`text-sm`) | 400 default, 700 active is NOT used — active = bg/color, not weight | ~1.4 | sidebar items, `TabsTrigger` |
| Body / answer text | 14px (`text-sm`) | 400 | 1.5 (`leading-relaxed`) | FAQ answers, tab content |
| FAQ question (summary) | 14–16px (`text-sm`/`text-base`) | 700 (`font-semibold`/`font-bold`) | ~1.4 | `<summary>` label |

**Hard rule (system-wide, enforced by CRM sidebar precedent):** never signal active/selected state with a heavier font weight. Active state = background + text color + icon tint only. Weights are 400 and 700 exclusively.

## Color

**Do not choose colors — inherit the shipped tokens.** All values are CSS variables in `app/globals.css`, per-palette + light/dark. Listed here so the checker can validate the 60/30/10 discipline the reorg must preserve.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (#f3ead8 light / #1a1714 dark — near-white/near-black, never pure) | Page canvas, main content area |
| Secondary (30%) | `--card` / `--secondary` (#fbf3e3 / #e9ddc4 light) | **Sidebar** (`bg-card`/`bg-sidebar`), cards, tab surfaces, nav hover (`hover:bg-secondary`) |
| Accent (10%) | `--primary` (pavette-driven, e.g. #d94a2b Rojo Forjo) | **Reserved for:** active sidebar row background (`bg-primary text-primary-foreground`), active tab underline/indicator, primary buttons ("Guardar cambios"), and the icon tint on the active nav row. |
| Destructive | `--destructive` (#b23a26 / #e05c43 dark) | Not introduced by this phase (no destructive actions here). Preserve existing usage in migrated tabs. |

Accent reserved for: **active sidebar nav row · active tab indicator · primary save buttons in migrated tabs.** Never apply `--primary` to group labels, inactive nav rows, FAQ chevrons, or the FAQ disclosure border. Bauhaus eyebrow primitives (`PageEyebrow`) use fixed constructivist hex (#d94a2b / #f4c543 / #2a5fa5), independent of palette — do not change.

**Contrast:** all inherited pairs already pass WCAG AA. New surfaces (FAQ disclosure) must use `text-foreground` on `bg-card`/`bg-background` and `text-muted-foreground` only for secondary/answer meta — verify ≥ 4.5:1.

## Interaction Contracts

### A. Grouped Sidebar (NAV-01)

**Structure (LOCKED by approved mock + D-02):** flat list → 5 labeled groups, in this order. Group→item mapping is authoritative:

| Group label | Items (menu keys → route) |
|-------------|---------------------------|
| **PANEL** | `dashboard` → /dashboard |
| **AGENDA** | `appointments` → /appointments · `agenda` → /agenda · `clients`/`patients` → /clients |
| **GESTIÓN** | `servicios` → /servicios · `equipo` → /equipo · `consultorios` → /consultorios · `negocio` → /negocio |
| **REPORTES** | `finances` → /finances |
| **AJUSTES** | `settings` → /settings |

- **MENSAJES group is excluded** (add-on dependency, out of milestone). Do not render it.
- **Vertical gating preserved (D-01):** groups are a data-driven array `{ section, keys[] }` inside `sidebar.tsx`, each key filtered against `resolveVertical(business).menu`. A group with zero surviving keys renders **nothing** (no empty header). Example: `canchas` has no `equipo` in its menu → the row disappears and GESTIÓN shows only its remaining items. Do NOT touch `lib/verticals.ts`.
- **Visual pattern (mirror `crm-sidebar.tsx`, do not import it):** group header = `--font-geist-mono` 11px uppercase `tracking-wider text-muted-foreground`, `px-2 pt-4 pb-1`. Items `space-y-0.5`.
- **Active state:** exact-path match (`pathname === item.href`). Active = `bg-primary text-primary-foreground` (preserve current dashboard sidebar's active treatment — it already uses `bg-primary`, do NOT switch to the CRM's `bg-secondary`+accent-bar unless matching the mock; **default = keep the existing dashboard active style** for behavior/visual continuity). Inactive = `text-muted-foreground hover:bg-secondary hover:text-foreground`.
- **States (all interactive rows):** default · hover (`hover:bg-secondary`) · focus-visible (`focus-visible:ring-[3px] ring-ring/50`, add if missing — accessibility non-negotiable) · active/current (`aria-current="page"`). No hover-only affordance.
- **Footer — behavior-frozen (D-03) + new FAQ link (D-07):** preserve "Ver mi página" (external link), "Cerrar sesión", and the "hecho con Forjo Studio" signature exactly. **Add** an "Ayuda" link in the footer region (above "Cerrar sesión" or with the other footer links), icon `HelpCircle` (lucide), same row styling as other nav rows, → `/ayuda`.
- **Mobile:** drawer behavior, overlay, `Menu`/`X` toggles — unchanged. Groups render identically in the drawer.

### B. Negocio Hub Tabs + Configuración split (NAV-02)

**Both rendered by the existing `SettingsClient` mega-component via `view` prop.** No new component. The migration = **reassign which `<TabsContent>` each view shows** (content of each tab is unchanged — behavior-frozen).

- **`/negocio` (view="negocio"):** becomes a hub. `TabsList` = **Datos del negocio · Cobros · Integraciones · Notificaciones/Mails** (in this order — "Datos" first per §3 brief; 4th tab labeled literally **"Notificaciones/Mails"** even though content is the current notificaciones tab). Tabs are **client-side state, NOT sub-routes** (D-04). Default active tab = "Datos del negocio".
- **`/settings` (view="config"):** `TabsList` reduces to **Apariencia · Seguridad · Suscripción** (drop Cobros/Integraciones/Notificaciones — they move to Negocio). Default active tab = "Apariencia".
- **Tab visual:** reuse `@/components/ui/tabs` exactly as today (`TabsList grid ... lg:flex lg:flex-wrap`, `TabsContent className="mt-4"`). Responsive: grid on mobile, wrap-flex on desktop (existing L809 pattern). Active tab indicator uses the shadcn tabs default (already themed).
- **No redirects (D-05):** Cobros/Integraciones/Notificaciones never had their own URL, so no legacy route to redirect. Both `/negocio` and `/settings` keep existing.
- **MP OAuth callback reroute (D-06) — part of behavior-frozen:** move the `?mp=connected|error` handling (toast + set Integraciones tab + `history.replaceState`) from `settings-client.tsx` L145 to wherever the Negocio hub renders, and change redirects `/settings?mp=...` → `/negocio?mp=...` in `mercadopago/callback/route.ts` (L16, L68) and `connect/route.ts` (L9). After OAuth, user must land on **Negocio → Integraciones tab** with the correct toast. `replaceState` target becomes `/negocio`.
- **`/negocio` page props:** the server component must now pass `secrets`, `mpConnectEnabled`, etc. (props the Cobros/Integraciones/Notificaciones tabs consume) — previously only Datos was shown.

### C. FAQ Disclosure Contract (HELP-01)

**Route:** `/ayuda` (Claude's discretion; `/ayuda` chosen over `/faq` — Spanish, user-facing). Server component `app/(dashboard)/ayuda/page.tsx` renders a static TS array `FAQ: { pregunta: string; respuesta: string }[]` (D-08 — versioned in git, no Supabase, no MDX).

**Header:** `PageEyebrow label="Ayuda"` + h1 "Ayuda" (`text-2xl font-bold font-[family-name:var(--font-heading)]`) + a one-line muted subtitle, e.g. "Cómo usar Forjo Gestión". Constrain body to `max-w-3xl` (matches settings, keeps optimal line measure 45–75 chars).

**Component:** **native `<details>`/`<summary>` disclosure** (zero-install), one per FAQ item, inside a `Card` or bordered list. Interaction contract (works identically if ever swapped for a real Accordion):

| Aspect | Contract |
|--------|----------|
| Expand/collapse | Click on `<summary>` toggles open. Multiple items MAY be open at once (independent disclosures — no forced single-open). |
| Summary row | `<summary>` = `flex items-center justify-between gap-3 px-4 py-3 cursor-pointer select-none list-none`, question in `font-semibold text-sm/text-base`. Remove default marker (`[&::-webkit-details-marker]:hidden` / `list-none`). |
| Chevron | lucide `ChevronDown`, `w-4 h-4 text-muted-foreground`, rotates 180° when open (`group-open:rotate-180 transition-transform`). Never accent-colored. |
| Answer | `px-4 pb-4 text-sm leading-relaxed text-muted-foreground` (or `text-foreground` if answers are long-form). Reveal is instant (no JS height animation required; CSS optional and ≤ 300ms if added). |
| Keyboard | `<summary>` is natively focusable + Enter/Space toggles (native behavior — do not override). Add visible `focus-visible:ring-[3px] ring-ring/50 rounded-lg` on the summary. |
| Dividers | `divide-y divide-border` between items, or `border-b` per row. Rounded container `rounded-lg border border-border`. |
| States | default · hover (`hover:bg-secondary/50` on summary) · focus-visible · open. |
| a11y | Semantic `<details>` provides native disclosure semantics — no ARIA needed. Headings inside answers must not skip levels (answers use `<p>`, not headings). |

**Two entry points (D-07):** (1) sidebar footer "Ayuda" link (see A); (2) a link/button from Configuración (`/settings`) — e.g. a small "¿Necesitás ayuda? Ver la guía" link, or an entry in the Seguridad/Apariencia surface footer. Planner picks placement; both must reach `/ayuda`.
