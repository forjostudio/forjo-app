# Phase 3 — UI Design Contract

> Visual and interaction contract for the **import de clientes por CSV** (DATA-03) of Forjo Gestión — a **4-step flow** (upload → preview/validación → confirmar → resumen) living in a single wide `Dialog` on `/clients`.
> Generated by gsd-ui-researcher.

**Scope note (read first):** This is a **UI surface over an already-shipped design system** (Bauhaus theme + shadcn `base-nova`/`neutral`, live in `app/globals.css`). It is **NOT a visual redesign**. This contract governs **LAYOUT · STRUCTURE · INTERACTION** of the 4-step import flow only. Colors, fonts, the spacing scale, and the 60/30/10 discipline are **inherited** from the Phase 1 and Phase 2 UI-SPECs (`../01-reorg-de-ia-ayuda/01-UI-SPEC.md`, `../02-alta-manual-exports-csv/02-UI-SPEC.md`) — **not re-chosen here**. Reuse-only: no new UI components, no new UI dependencies (`papaparse` is server-side parsing logic, not UI). UI fidelity + visual consistency with sibling components (the alta/merge dialogs) is load-bearing for this developer.

**Container decision (RESOLVED with user — Opción A):** the whole flow lives in **one wide `Dialog`** (`DialogContent className="sm:max-w-2xl"`, ~672px). The 4 stages swap the Dialog **content** (upload → preview → resumen), not the surface. This is pure reuse of the dominant pattern on `/clients` (alta + merge are already Dialogs), **zero new routes**. On mobile (375px) the preview renders as **stacked card-like rows** (`divide-y`) instead of a table; the real `Table` (reusing `components/ui/table.tsx`) kicks in from `sm:` up.

## Spacing Scale

Inherited (all multiples of 4 — matches existing Tailwind v4 usage in `clients-client.tsx`). This phase declares only the values it uses.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (`gap-1`, `p-1`) | Icon-to-label micro-gaps inside counter badges; tight stacking |
| sm | 8px (`gap-2`, `space-y-2`, `p-2`) | Counter row gap; dialog footer button row; dropzone inner icon/text stacking; header action-row gap (existing `gap-2`) |
| md | 12px (`gap-3`, `px-3`, `py-3`, `space-y-3`) | Preview row inner padding (mobile stacked rows), header vertical rhythm (existing `space-y-3` / `p-4` sub-blocks) |
| md+ | 16px (`p-4`, `space-y-4`) | Dialog content padding (`DialogContent` default `p-4`), stage vertical rhythm, dropzone outer padding |
| lg | 24px (`p-6`, `py-6`) | Dropzone idle target vertical padding (generous drop area), resumen block padding |

**Hard rules (from scope note):**
- **Only multiples of 4.** Do NOT introduce `gap-1.5` / `p-1.5` / `space-y-1.5` on the new surfaces. The existing header uses a stray `gap-1.5` on its buttons (harmless legacy) — **do not copy it onto the new "Importar CSV" button**; rely on the `Button` component's own built-in gap.
- **`Button` built-in gap:** the shadcn `Button` (`components/ui/button.tsx`) ships its own icon-to-label gap in the size variant. **Never set a manual `gap-*` on a `Button`.** Icon (`w-4 h-4`) + label render with the component's own spacing.

**Exceptions:**
- **Touch targets ≥ 44px:** the "Importar CSV" trigger button and the primary "Confirmar import" / "Cerrar" actions must present a ≥ 44px effective hit area on 375px mobile. Use `Button` default size and let the row/padding bring the hit area to ≥ 44px; do NOT use icon-only variants for these actions (labels required — see Copywriting).
- The dropzone is a large intentional target (`min-h` ~`py-6`/`py-8`), well above 44px.
- Counter `Badge`s reuse the fixed `h-5` (20px) — they are **non-interactive status indicators**, so the 44px rule does not apply.

**Do not choose colors — inherit the shipped tokens** (CSS variables in `app/globals.css`, per-palette + light/dark). Listed so the checker can validate the 60/30/10 discipline.

| Role | Token | Usage in this phase |
|------|-------|---------------------|
| Dominant (60%) | `--background` (near-white / near-black, never pure) | Dialog surface base, dropzone idle fill, table body rows (valid rows) |
| Secondary (30%) | `--card` / `--secondary` / `--muted` | Table header band, dropzone border (`border-dashed border-border`), duplicated-row muted treatment, the neutral resumen tiles, the "Importar CSV" trigger (`outline`) and "Cancelar"/"Cerrar" (`outline`) buttons |
| Accent (10%) | `--primary` (palette-driven, e.g. Rojo Forjo) | **Reserved for:** the **primary "Confirmar import" CTA** only (default Button = `bg-primary`). Nothing else in this flow gets `--primary`. |
| Destructive | `--destructive` | **Error rows in the preview** (border/fill + text + icon), the per-row error message (`text-destructive`), the invalid-file message, and the "N con error" / "fallidos" counter. Destructive here means **validation error**, not a destructive action — there is no delete in this flow. |

**Accent reserved for:** the **"Confirmar import"** primary button only. Never apply `--primary` to: the "Importar CSV" trigger (outline), the dropzone, valid preview rows, the counters, "Cancelar"/"Cerrar", or the resumen tiles.

### Preview error-row treatment (SC-1 — decided, not asked)

Rows with a validation error (nombre faltante, sin contacto, teléfono no numérico) must be **unmistakably marked before anything is written**. The design system constrains the answer (reuse-only, WCAG AA, do not rely on hue alone):

| Row class | Visual treatment (desktop `Table`) | Visual treatment (mobile stacked row) | Why |
|-----------|-----------------------------------|----------------------------------------|-----|
| **Válida** (importable) | Default row on `--background`. Optional leading `CheckCircle2` in `text-muted-foreground` (quiet). | Card row on `--background`, name + contact stacked. | The common, calm state — no attention spend. |
| **Con error** (validación) | `bg-destructive/10` row fill + `border-l-2 border-destructive` (or `text-destructive` on an `AlertCircle` leading icon) + the error message in `text-xs text-destructive` in the "Estado" cell. | Card row with `border-l-2 border-destructive` + `bg-destructive/10` + inline error message below the name. | Destructive band; distinguished by **fill + border + icon + text**, not hue alone → survives color-blindness and passes WCAG AA (inherited tokens). |
| **Duplicada** (se omitirá) | `text-muted-foreground` (dimmed) row + a `Badge variant="secondary"` "Duplicado" in the Estado cell. Optional `opacity-70`. | Dimmed card + "Duplicado" secondary badge. | Neither error nor import — it is a **skip**. Muted/secondary band (30%), visually distinct from both valid (default) and error (destructive). |

- The three row classes differ by **fill + border + icon + a text label**, never by color alone → distinguishable without color perception.
- Error rows are **non-blocking**: they render inline in the preview and are simply **excluded** from the import (they land in "fallidos"). The user is NOT forced to fix them (v1) — the preview just shows what will and won't import.

### Counters (SC-1) — color mapping

| Counter | Look | Token |
|---------|------|-------|
| **N válidas** | `Badge variant="outline"` or plain `text-foreground` | neutral (does not compete) |
| **N con error** | `Badge variant="destructive"` (`bg-destructive/10 text-destructive`) | destructive band |
| **N duplicadas** | `Badge variant="secondary"` | secondary band |

### Resumen tiles (SC-4) — color mapping

Three side-by-side tiles (`Card` or a simple `divide-x`/grid block), **neutral by default** — the resumen is informational, not a CTA:

| Tile | Number color | Rationale |
|------|--------------|-----------|
| **Importados** | `text-foreground` (or a subtle `CheckCircle2` in `text-muted-foreground`) | Success is the expected outcome; keep it calm, do NOT spend accent on a passive stat. |
| **Omitidos** (duplicados) | `text-muted-foreground` | A neutral skip. |
| **Fallidos** (validación) | `text-destructive` **only if > 0**, else `text-muted-foreground` | Draw the eye to failures only when there are any. |

**Contrast:** all inherited pairs pass WCAG AA. New surfaces (dropzone, table, resumen tiles) use `text-foreground` on `bg-background`/`bg-card`, `text-muted-foreground` for helper text, `text-destructive` for errors — inherited defaults already pass ≥ 4.5:1.

## Header placement (new "Importar CSV" button)

The `/clients` left-panel header (`clients-client.tsx` L500–528) is a narrow `lg:w-80` column with `space-y-3`. After Phase 2 it holds: row 1 (título + merge icon), row 2 (`grid grid-cols-2`: "Exportar CSV" outline + "Nuevo cliente" primary). Adding a third button must not clip in the narrow panel.

**Decided layout (RESOLVED with user):**
- **Row (secondary, 2-col grid):** **"Exportar CSV"** (`outline`) + **"Importar CSV"** (`outline`) — the two CSV round-trip actions paired, both secondary weight. `grid grid-cols-2 gap-2`, each `w-full`.
- **Row (primary, full-width):** **"Nuevo cliente"** (`Button` default = `bg-primary`, full-width `w-full`) as its own row so the primary CTA stays dominant and uncrowded.
- The exact row order (primary above or below the CSV pair) is executor discretion, but **the primary CTA must not sit in the same 2-col grid as the two secondary CSV buttons** (that would make three cramped cells at 320px and dilute the primary).

**Button spec for "Importar CSV":**
- `variant="outline"`, leading icon lucide **`Upload`** (`w-4 h-4`), label **"Importar CSV"**.
- **No manual `gap-*`** — the `Button` component supplies the icon-to-label gap. (The legacy `gap-1.5` on the sibling export `<a>` is not to be replicated on the new button.)
- Opens the import `Dialog` (`setImportOpen(true)`).
- ≥ 44px effective hit area on mobile.

## Interaction Contracts

The flow is **one `Dialog`** whose content advances through **4 stages** driven by local React state (e.g. `stage: 'upload' | 'preview' | 'confirming' | 'resumen'`). The `Dialog` surface, title, and footer persist; the body swaps.

**Dialog shell:**
- `DialogContent className="sm:max-w-2xl"` (~672px; wider than the alta's `sm:max-w-md` to let the preview table breathe). On mobile it fills `max-w-[calc(100%-2rem)]` (the DialogContent default).
- `DialogTitle` = **"Importar clientes"** (or `Importar ${term.clients.toLowerCase()}` to respect vertical terminology, e.g. "Importar pacientes" en salud).
- Native `Dialog` close (X button, Escape, click-outside) is available in **upload / preview / resumen**. **Disable close-on-outside/Escape during `confirming`** (write in flight) to avoid abandoning a half-written import — re-enable when the response lands.
- Footer (`DialogFooter`) holds the stage's actions (see per-stage below).

### Stage 1 — Upload

| Aspect | Contract |
|--------|----------|
| Layout | A **dropzone**: a large centered target, `border-2 border-dashed border-border rounded-lg`, generous padding (`py-6`/`py-8`, `px-4`), containing a leading `Upload` icon (`w-8 h-8 text-muted-foreground`), a one-line instruction, and an **"Elegir archivo"** `Button` (`outline`) that triggers a **visually-hidden native `<input type="file" accept=".csv,text/csv">`**. |
| States | **idle** (dashed border, muted instruction) · **dragover** (if drag&drop wired: `border-primary`/`bg-muted` to signal a valid drop target — drag&drop is optional, but a plain click-to-choose file input is the required baseline) · **file selected** (show the chosen file name + size + a small icon-only `X` to clear, replacing the instruction; the `X` MUST carry `aria-label="Quitar archivo"` + `focus-visible` ring — icon-only a11y, checker Dim-2 FLAG) · **loading** (after "Continuar" → preview: `Loader2` spinner + disabled, while the `preview` route parses server-side) · **error** (invalid file, see below). |
| Client-side guard | On file select, validate **extension `.csv`** and **size ≤ 2MB** client-side for instant feedback; the server re-validates authoritatively (D-03/D-04). If invalid, do NOT advance — show the error message inline under the dropzone and keep the dialog on Stage 1. |
| Advance | A primary-intent action **"Continuar"** (or the file-select itself auto-POSTs to `preview` — executor discretion; if a "Continuar" button is used it is the accent CTA of this stage). POSTs the file (`multipart/form-data`) to the `preview` route handler; on `ok` → Stage 2 with the returned counts + error rows. |
| Empty / no file | "Continuar" is disabled until a valid `.csv` is chosen. |
| Error (invalid file) | Inline message in `text-sm text-destructive` under the dropzone: **"Subí un archivo .csv de hasta 2 MB."** (covers wrong extension AND oversized). Keep Stage 1. |
| Error (bad header) | If the server rejects the header (not the canonical `nombre,telefono,email,origen,notas,obra_social,nro_obra_social`), inline message: **"El CSV no tiene el formato esperado. Exportá tus clientes desde Forjo y usá ese archivo como base."** Keep Stage 1. |
| Error (network) | `toast.error('No se pudo procesar el archivo. Intentá de nuevo.')`, re-enable "Continuar", keep Stage 1. |

**File retained in browser state:** the selected `File` is held in React state so it can be **re-posted to `confirm` without re-selecting** (stateless re-upload per RESEARCH Pattern 4 — the user never re-picks the file; "Confirmar import" re-sends the same `File`).

### Stage 2 — Preview (SC-1 — nothing is written)

| Aspect | Contract |
|--------|----------|
| Header of stage | A one-line summary + the **counters** row: `Badge`s for **"N válidas"** (neutral/outline), **"N con error"** (destructive), **"N duplicadas"** (secondary). Counters come from the `preview` response (`importables`, `errores.length`, `duplicadas`). |
| Table (desktop, `sm:` up) | Reuse `components/ui/table.tsx`. Columns: **Nombre · Teléfono · Email · Estado**. The "Estado" cell shows nothing for valid rows, the **error message** (`text-xs text-destructive`) for error rows, and a **"Duplicado"** `Badge variant="secondary"` for duplicated rows. Row treatment per the **Preview error-row treatment** table above. Table body is scrollable (`max-h-[50vh] overflow-y-auto`) so a 2.000-row CSV does not blow out the dialog. |
| List (mobile, `< sm:`) | **Stacked card-like rows** (`divide-y divide-border`), NOT the table. Each row: name (primary line `text-sm`), contact (secondary line `text-xs text-muted-foreground`), and — for error/duplicate rows — the `border-l-2` + fill + inline error/"Duplicado" badge. Same `max-h`/scroll treatment. |
| Row numbering | Error rows may show the source line number (`Fila N`, where N matches Excel: header = 1) as a muted prefix, so the owner can find the row in their spreadsheet. Optional but recommended. |
| Nothing written | This stage is **pure UX**. No write happens. The server already re-validates on confirm (D-03) — the preview is not authoritative. |
| Empty state (no valid rows) | If `importables === 0` (all rows errored or the CSV had no data rows): show an empty-state block (icon `AlertCircle` + heading + body), and **disable "Confirmar import"**. Heading: **"No hay filas para importar."** Body: **"Revisá que el CSV tenga la primera fila de encabezados y al menos un cliente con nombre y un contacto (teléfono o email)."** Offer **"Elegir otro archivo"** (goes back to Stage 1). |
| Back | A **"Cambiar archivo"** / "Atrás" (`ghost` or `outline`) action returns to Stage 1 (keeps the dialog open). |
| Confirm | Primary `Button` **"Confirmar import"** (`bg-primary`) — the single accent spend of the flow. Enabled only when `importables > 0`. Optionally show the count in the label: **"Importar N clientes"**. |
| States | "Confirmar import": default · hover · focus-visible · disabled (when `importables === 0`). Table rows: valid · error · duplicate (per the treatment table). |

### Stage 3 — Confirmar (write in flight)

| Aspect | Contract |
|--------|----------|
| Trigger | Click "Confirmar import" → `setStage('confirming')`, **disable the button** and show label **"Importando..."** with a leading `Loader2` spinner (`animate-spin w-4 h-4`). Anti-double-submit is mandatory (the write is not idempotent — reuse the existing `saving`-disable pattern from the alta modal). |
| Re-upload | Re-POST the **same `File`** (held in browser state) as `multipart/form-data` to the `confirm` route handler. The server re-parses + re-validates + re-dedups + inserts (D-03). The client does NOT send "the rows it saw". |
| Dialog lock | During `confirming`, **disable close-on-outside and Escape** (do not let the user abandon a write mid-flight). Re-enable when the response returns. |
| Success | On `ok` with `resumen` → advance to Stage 4 (Resumen). |
| Error (network / server) | `toast.error('No se pudo completar la importación. Intentá de nuevo.')`, revert to Stage 2 (preview) with the button re-enabled, re-enable close. Do NOT lose the loaded file/preview. |

### Stage 4 — Resumen (SC-4)

| Aspect | Contract |
|--------|----------|
| Layout | Three side-by-side count tiles (`grid grid-cols-3 gap-2` on `sm:`, stacked `grid-cols-1` or a compact row on mobile). Each tile: a big number (`text-2xl font-bold`) + a small label (`text-xs`). From the `confirm` response `resumen`. |
| Tiles | **Importados** (`resumen.importados`) · **Omitidos** (`resumen.omitidos`, duplicados) · **Fallidos** (`resumen.fallidos`, validación). Number colors per the **Resumen tiles color mapping** above. |
| Headline | A one-line confirmation above the tiles: **"Importación completada."** (or `${resumen.importados} clientes importados.`). |
| Note (optional) | If `fallidos > 0` or `omitidos > 0`, a muted helper line: **"Las filas con error no se importaron; las duplicadas se omitieron."** |
| Primary action | **"Cerrar"** (`Button`, can be `bg-primary` OR `outline` — since the flow is done, `outline`/neutral is acceptable and keeps accent reserved; executor discretion but prefer neutral). Closes the dialog and **reflects the new clients in the list**: the parent must re-fetch or optimistically insert the imported clients so they appear immediately with the **"Importado"** origin badge (SC-4, consistent with the alta's instant-insert behavior). |
| Live list update | On close, the `/clients` list must show the newly imported clients (badge **"Importado"** — the `secondary` variant already defined in the Phase 2 UI-SPEC). Prefer a re-fetch of the clients list over manual state surgery, given batch size. |
| Reset | Closing resets the flow state (stage → 'upload', clear the held `File`, clear counts) so a second import starts clean. |

## Behavior-Frozen Guardrails (for executor)

- Do NOT modify the two-panel `/clients` layout beyond inserting the new "Importar CSV" button and the import `Dialog`.
