# Phase 2 — UI Design Contract

> Visual and interaction contract for the onboarding UX rework. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Nature:** This is a UX rework of an EXISTING screen (`app/(onboarding)/onboarding/page.tsx`), NOT a new design. The project has a locked design system. This spec DOCUMENTS the existing tokens/components the rework must respect, and specifies DELTA contracts only for the changes in CONTEXT.md decisions D-01..D-09. No new palette, typography scale, component set, steps, or fields.

---

## Scope of Change (delta contracts only)

| # | Change | Decision | Contract section |
|---|--------|----------|------------------|
| 1 | "Omitir" button on non-mandatory steps (advances to next step; only step 1 is mandatory) | D-01, D-02, D-04 | §Interaction — Omitir button |
| 2 | Dynamic stepper: "Profesionales" auto-hidden for `canchas` vertical (3 steps instead of 4) | D-03 | §Interaction — Dynamic stepper |
| 3 | Servicios step: fixed column header (labels always visible, not only first row) | D-07 | §Interaction — Servicios column header |
| 4 | Inline validation onBlur with immediate feedback on formatted fields (WhatsApp, price) | D-08 | §Interaction — Inline validation |
| 5 | Allow price 0 in a completed service row | D-09 | §Interaction — Servicios validation |

**Out of scope (do NOT touch):** full visual redesign, new steps/fields, step order (D-06 keeps Negocio → Servicios → Profesionales → Horarios), the schedule engine / `time_blocks` write path (closed in Phase 1), `handleFinish` insert filtering (already discards empties).

## Color

Bauhaus dark/cream neutrals + a single per-business accent (`--primary`, swapped by `data-palette`). All values are TOKENS, never hardcoded hex. The rework references tokens only.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (`#f3ead8` light / `#1a1714` dark) | Page background |
| Secondary (30%) | `--card` (`#fbf3e3` / `#252019`), `--secondary` (`#e9ddc4` / `#2e2820`) | Onboarding Card surface, inactive stepper nodes, inactive day toggles |
| Accent (10%) | `--primary` (per-palette, default `#d94a2b`) | See reserved list below |
| Destructive | `--destructive` (`#b23a26` / `#e05c43`) | Inline validation errors, remove/trash hover |
| Positive | `text-green-500` (existing) | Slug "✓ Disponible" only |

**Accent (`--primary`) reserved for (explicit — never "all interactive elements"):**
- Active/completed stepper node (`bg-primary text-primary-foreground`, active node adds `ring-4 ring-primary/20`).
- Completed stepper connector line (`bg-primary`).
- Primary CTA button ("Siguiente" / "Finalizar y entrar al dashboard") — shadcn default `Button` variant.
- Enabled day toggle in Horarios (`bg-primary text-primary-foreground`).
- Vertical hint callout border/background (`border-primary/30 bg-primary/5`) and its icon.
- Focus ring on all interactive controls (`--ring` = `--primary`).

**"Omitir" button MUST NOT use accent.** It is a de-emphasized secondary action → `variant="ghost"` (text/link weight), so the accent stays reserved for the forward primary CTA (one dominant action per screen). Destructive token is NOT used for "Omitir" (skipping is not destructive).

Contrast: all token pairs already meet WCAG AA in both modes (Bauhaus palette is contrast-tuned). New error text uses `text-destructive` on card surface (passes AA). Never pure `#000`/`#fff`.

## Interaction Contracts (the rework)

### Omitir button (D-01, D-02, D-04)

- **Where:** In the existing nav bar (`page.tsx` ~line 659), only on non-mandatory steps (2 Servicios, 3 Profesionales, 4 Horarios). Step 1 (Negocio) shows NO Omitir.
- **Placement:** In the right cluster next to the primary CTA. Layout: `Atrás` (left) — `[ Omitir por ahora ]  [ Siguiente ]` (right). "Omitir por ahora" sits immediately left of "Siguiente" as the lower-emphasis sibling.
- **Style:** `variant="ghost"` (link/text weight, no accent fill). Never destructive, never primary. Keeps one dominant action (Siguiente/Finalizar) per screen.
- **Behavior:** advances to the next step (`setStep(s => s + 1)`), NOT skip-to-end. It does NOT run the step's validation (skipping bypasses the optional fields). Empties are already filtered at `handleFinish` (D-05) — no data-model change.
- **Last step:** on the last step there is no "Omitir por ahora"; the primary CTA becomes "Finalizar y entrar al dashboard". Finishing with the last step blank is valid (Horarios/Profesionales left empty).
- **States (all required):**
  - default: ghost, `text-foreground`/`text-muted-foreground`.
  - hover: `hover:bg-accent hover:text-accent-foreground` (shadcn ghost default) — subtle, no accent fill.
  - focus-visible: visible ring `focus-visible:ring-2 focus-visible:ring-ring` (shadcn default). Non-negotiable.
  - active: shadcn default pressed.
  - disabled: not applicable (Omitir is always enabled on optional steps).
- **Mobile (375px):** on narrow widths the right cluster may wrap; keep Omitir and Siguiente both tappable (full available width if stacked), Omitir above/left of Siguiente so the primary action is the last/most prominent.

### Non-mandatory step gating relax (D-02)

- `canGoNext` keeps enforcing only step 1 (`name && slug && slugAvailable && type`). Steps 2/3/4 no longer block "Siguiente" (drop the `services.every(...)` / `professionals.every(...)` gates). "Siguiente" on optional steps is always enabled; the user may fill or skip.

### Dynamic stepper — canchas auto-hide (D-03)

- The `steps` array becomes vertical-dependent: filter out "Profesionales" when `getVerticalKeyByType(type) === 'canchas'`. Result: canchas = 3 steps (Negocio → Servicios → Horarios); all other verticals = 4 steps.
- **Numbering consistency:** step numbers/indices are derived from the FILTERED array position, not hardcoded `n`. The visible node numbers must read 1-2-3 (canchas) / 1-2-3-4 (others) with no gaps. The "last step" (where Finalizar appears) is the last item of the filtered array (Horarios in both cases).
- Stepper visual contract unchanged: `w-9 h-9` rounded node, active = `bg-primary ... ring-4 ring-primary/20`, completed = `bg-primary` + `Check` icon, pending = `bg-secondary text-muted-foreground`; connector `bg-primary` (done) / `bg-border` (pending). Caption `text-xs` hidden below `sm`.
- Header subtitle "Configurá tu negocio en 4 pasos" must reflect the actual count (e.g. "en 3 pasos" for canchas) or be made count-agnostic ("en pocos pasos").

### Servicios column header (D-07)

- Replace the per-first-row labels (`{i === 0 && <Label>}`) with a SINGLE fixed header row above the service rows, aligned to the existing 12-col grid (`grid grid-cols-12 gap-2`): `Nombre` = `col-span-5`, `Min.` (Clock icon) = `col-span-3`, `Precio` (DollarSign icon) = `col-span-3`, trash spacer = `col-span-1`.
- Header labels: `text-xs text-muted-foreground`, same as today's first-row labels. Always visible regardless of which row has focus. Do NOT make it a sticky header (short form; a fixed non-sticky header row is enough — matches discretion note).
- Service rows lose their inline labels (now redundant); row grid/spacing unchanged (`items-end` → becomes `items-center` is acceptable since labels no longer offset row 1, but keep vertical alignment consistent across rows).

### Inline validation onBlur (D-08)

- Extend the existing inline-error pattern (`validateHours` per-block, `b.error` → `<p class="text-xs text-destructive">`, `aria-invalid` on inputs) to the formatted fields:
  - **WhatsApp (step 1):** onBlur, if non-empty and `normalizeArWhatsApp(whatsapp)` returns null → set inline error + `aria-invalid`; clear on change/valid. Message per copy contract. (WhatsApp stays optional — empty is valid, no error.)
  - **Precio (Servicios):** onBlur, if value < 0 → inline error + `aria-invalid`; price 0 and positive are valid (D-09). Clear on change.
- **Error display contract:** message in `text-xs text-destructive` directly under the field (block-level `<p>`); input carries `aria-invalid={!!error}`. Feedback is immediate onBlur, not deferred to Siguiente/Finalizar. Mirror the block-error markup already in the Horarios step (`page.tsx` ~line 637) and the panel reference (`agenda-client.tsx:validateBlocks`).
- Slug already validates live (debounced availability check) — keep as-is; it's the existing model of immediate feedback.

### Servicios validation — allow price 0 (D-09)

- Empty rows are ignored (filtered at `handleFinish`: `services.filter(s => s.name)`).
- A row WITH a name is accepted with price ≥ 0 (0 = free service, e.g. consulta gratuita). Only negative price is invalid (onBlur inline error above).
- Remove the old `price > 0` requirement from any advance gating (already relaxed by D-02, but ensure no residual `price > 0` check blocks a valid free service). Confirm `services` insert and the panel tolerate price 0 (per D-09 verification note — a planner/executor concern, flagged here).

## Accessibility contract (global rules applied)

- Mobile-first 375px: all steps already use `grid-cols-1 sm:grid-cols-2` stacking; keep. Nav bar stays usable stacked.
- Focus visible on ALL interactive controls (`focus-visible:ring-2 focus-visible:ring-ring`) — includes the new Omitir button and Servicios inputs. Never remove focus outlines.
- Inputs with errors expose `aria-invalid`; toggle buttons expose `aria-pressed` (existing palette/day pattern — keep).
- Heading hierarchy: page brand (visual, not an h-tag) → step `<h2>`. Do NOT introduce an `<h1>`-skip; steps stay `<h2>`. No new heading levels.
- Contrast AA in light AND dark (Bauhaus tokens are pre-tuned); error `text-destructive` on `--card` passes.
