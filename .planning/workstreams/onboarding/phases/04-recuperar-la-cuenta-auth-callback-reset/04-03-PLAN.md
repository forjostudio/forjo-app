    Creá `components/auth/check-your-email.tsx` (carpeta nueva), componente `'use client'` con **named export**
    `CheckYourEmail`. Se usa **dos veces**: en `/forgot-password` (D-02) y en el card de `/register` (D-12) — son
    la misma pantalla conceptual ("andá a tu mail"), por eso se comparte en vez de duplicarse.

    **Props:**
    - `email: string` — la dirección a mostrar.
    - `title: string` — el encabezado (cada uso pone el suyo).
    - `description: React.ReactNode` — el cuerpo del mensaje (cada uso pone el suyo).
    - `onResend: () => Promise<void>` — dispara el reenvío. El componente **no sabe** qué se reenvía.
    - `children?: React.ReactNode` — el link secundario del pie (cada uso pone el suyo).

    **Constante exportada:** `RESEND_COOLDOWN_SECONDS = 60` (D-05).

    **Mecánica del cooldown (D-05 — discreción resuelta acá):**
    - Estado `secondsLeft: number`, inicializado en `RESEND_COOLDOWN_SECONDS`. **Arranca bloqueado apenas monta**,
      no en 0: cuando este componente aparece, el primer mail **acaba de salir**. Si arrancara en 0, el usuario
      clickea "Reenviar" al toque y quema la cuota — que es exactamente lo que D-05 evita.
    - Un `useEffect` con `setInterval` de 1000ms que decrementa hasta 0 y ahí limpia el intervalo. Cleanup
      obligatorio: `let cancelled = false` en el scope del efecto y `return () => { cancelled = true; clearInterval(id) }`,
      igual que `plan-banner.tsx:44-61`. Sin eso, el intervalo sobrevive al unmount y setea estado sobre un
      componente muerto.
    - `handleResend`: si `secondsLeft > 0` no hace nada; si no, setea un `resending` local para deshabilitar el
      botón, hace `await onResend()`, y **pase lo que pase** (incluido error) resetea `secondsLeft` a
      `RESEND_COOLDOWN_SECONDS` en un `finally`. Motivo de seguridad, no de UX: si el cooldown solo se reiniciara
      en el camino feliz, un error de rate limit dejaría el botón habilitado y **el propio rate limit se
      convertiría en un oráculo lateral de enumeration** (Pitfall 4 del RESEARCH).
    - Botón: cuando `secondsLeft > 0` → `disabled` y texto **`Reenviar en {secondsLeft}s`** (el literal de D-05).
      Cuando llega a 0 → habilitado y texto **`Reenviar mail`**. Mientras `resending` → `Reenviando...`.
      Variante `outline` (es acción secundaria; el CTA de la pantalla ya se ejecutó).

    **Copy y estructura (mobile-first, dentro del `max-w-[340px]` que aporta el layout):**
    - `<h1>` con el `title` recibido, misma clase que el h1 del login
      (`font-[family-name:var(--font-heading)] text-2xl font-bold`).
    - `<p>` con el `description` recibido, misma clase que el subtítulo del login
      (`text-muted-foreground text-sm mt-1.5 mb-6`).
    - El `email` se renderiza **dentro del `description`** que pasa cada consumidor (así cada pantalla decide su
      frase), pero el componente igual recibe `email` como prop para poder usarlo en el `aria-label` del botón de
      reenvío (`Reenviar mail a {email}`).
    - Botón de reenvío `w-full`.
    - `{children}` al final, para el link del pie.

    **Restricción de seguridad NO negociable (D-02 / D-14 / Pitfall 4):** este componente **no tiene ninguna rama
    condicional que dependa de si la cuenta existe**, y no recibe ningún prop que lo permita. Renderiza siempre lo
    mismo. Poné un comentario de cabecera en español que lo diga: si el mensaje variara según exista o no la
    cuenta, el flujo se convertiría en un **oráculo de quién tiene cuenta en Forjo** (user enumeration). El
    componente es el lugar donde esa garantía se hace estructural — no depende de que cada página se acuerde.

    Accesibilidad: el contador que cambia cada segundo no debe spamear lectores de pantalla → **no** le pongas
    `aria-live` al texto del botón.
