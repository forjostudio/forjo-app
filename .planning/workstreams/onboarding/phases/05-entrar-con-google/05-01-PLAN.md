<objective>
La única extensión de código en el callback endurecido de Phase 4: una rama que detecta el retorno de
OAuth por la presencia de `?code=` y lo canjea con `exchangeCodeForSession`, más una rama de error OAuth
propia para el fallo del canje y la cancelación del consent. El path de mail (`token_hash` + `verifyOtp`)
queda **intacto**.

botón de Google (plan `05-02`) manda al usuario a `/auth/callback?code=...`, ese `code` cae en el parse de
`token_hash` de Phase 4, falla, y el usuario termina en `/forgot-password?error=invalid_link` — un mensaje
de recuperación de contraseña que no tiene nada que ver (Pitfall 1/3 del research).
Output: `lib/auth/callback.ts` (1 fila nueva), `lib/auth/callback.test.ts` (test + comentarios corregidos),
`app/auth/callback/route.ts` (la rama nueva).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google / GoTrue → `/auth/callback` (anónimo) | El retorno de OAuth trae `?code=` (o `?error=`); un usuario anónimo lo alcanza. El `code` autoriza el canje por sesión. |
| `/auth/callback` → GoTrue | `exchangeCodeForSession` lee el `code_verifier` de la cookie y setea las cookies de sesión; acá ocurre el auto-link de identidad. |
| Query del callback → redirect | Todo lo que viene en la URL (`code`, `error`) es input NO confiable; el destino se deriva server-side, nunca se refleja. |
