| Boundary | Description |

| Navegador (login/register) → GoTrue `/authorize` | El botón inicia el round-trip OAuth; el `redirectTo` que viaja debe estar en la allowlist del Dashboard. |
| Query de `/login` → render | `?error=oauth` es input no confiable; se compara contra un literal fijo, nunca se refleja. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02 | Tampering | Open redirect vía el `redirectTo` de `signInWithOAuth` | mitigate | `redirectTo` hardcodeado a `${window.location.origin}/auth/callback` — nunca un input del usuario ni el destino final. El destino real se deriva server-side en el callback (plan `05-01`). Debe estar en la allowlist sin wildcard del Dashboard (D-06, verificado en `05-03`). Verificado por `grep -c "window.location.origin" >= 1`. |
| T-05-03 | Spoofing | CSRF del round-trip OAuth | mitigate | El `state` anti-CSRF lo genera y valida GoTrue; el botón no lo toca ni lo reinventa. Delegado por diseño del proveedor. |
| T-05-07 | Repudiation / UX | El usuario ve un error opaco al fallar/cancelar Google (viola D-05) | mitigate | `/login` lee `?error=oauth` y muestra un mensaje claro con la salida ("probá de nuevo, o entrá con tu email"). `role="status"`, tokens del design system, nunca una pantalla muerta ni un toast genérico. Verificado por `grep -c "No pudimos entrar con Google" == 1` en login. |
