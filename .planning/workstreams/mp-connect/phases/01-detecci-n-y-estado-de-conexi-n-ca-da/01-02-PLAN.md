<objective>
Terminar con el fallback mudo del resolver de token de MercadoPago Connect: cuando MP rechaza el refresh o falla la persistencia del token rotado single-use, la conexión se trata como caída (se marca `'error'` y el resolver devuelve `null` — no se cobra con un token vencido/no persistido); un refresh exitoso auto-sana el flag; un 401 en el cobro marca la conexión caída; y la reconexión OAuth limpia el flag. Todo fallo queda logueado server-side con el motivo real.

Purpose: Es el corazón del milestone (integridad de pagos + rotación single-use del refresh). Hoy `getValidMpAccessToken` devuelve el token vencido ante un refresh rechazado (payment.ts §47-50) y no chequea el error de persistencia (§54-61) → el `refresh_token` single-use se consume y la conexión queda rota sin aviso. Este plan cierra ambos huecos y suma la detección por 401 (D-04) y el auto-heal (D-05).
Output: `getValidMpAccessToken` y `createDepositPreference` endurecidos en `lib/payment.ts`; heal en el callback OAuth; tests del resolver, del 401 y del heal.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente final (booking) → route handler del cobro | El slug/appointmentId llegan del cliente; el business_id se resuelve server-side. El flag se escribe con ese business_id resuelto, nunca con un id del cliente. |
| dueño autenticado (callback OAuth) → businesses | El callback usa el session client owner-scoped (`.eq('owner_id', user.id)`), RLS activa: sana solo el negocio del dueño. |
| resolver server-side → business_secrets / businesses (service-role) | El refresh persiste tokens y marca el flag con service-role; el filtro por business_id es la única barrera de aislamiento. |
