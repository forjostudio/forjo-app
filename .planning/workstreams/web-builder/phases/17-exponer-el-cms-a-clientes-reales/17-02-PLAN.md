<objective>
Cerrar la 4a superficie de escritura del CMS (el UPLOAD de imagenes) con `has_web_custom`. Hoy el editor de Phase 14 sube DIRECTO a Supabase Storage desde el browser (session client), y la unica barrera es la RLS del bucket `landing-assets`, que gatea `owner_id` pero NO el add-on: un dueno sin `has_web_custom` puede subir objetos a su propio prefijo. Este plan agrega `AND has_web_custom = true` al subquery de las policies INSERT + UPDATE del bucket (migracion 051), verifica el trigger anti-tampering y prueba el gate contra Storage hosteado. Completa la parte de datos de **PUB-01 / SC2**.

Purpose: al retirar el kill-switch global (17-01), el gate unico `has_web_custom` tiene que sostener en las 4 superficies del CMS — guardar, publicar, descartar y **upload**. Las 3 primeras ya lo gatean en la Server Action; el upload no pasa por Server Action, asi que su unico gate no-bypasseable es la policy RLS.

Output: `supabase/migrations/051_landing_assets_gate_entitlement.sql` + extension del test de aislamiento + verificacion funcional del gate contra Storage prod-like.
</objective>
