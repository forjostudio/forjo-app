<objective>
Agregar el BACKSTOP DE INTEGRIDAD declarativo de ESPACIO-03: la tabla de proyección `appointment_spaces` (una fila por turno×espacio) con su propio EXCLUDE gist y los triggers que la pueblan/limpian desde `appointments`. Es la única red de seguridad real si la derivación del advisory lock del Plan 01 tuviera un bug — un EXCLUDE declarativo y atómico que ningún error de app-logic puede bypassear.

> RECORTABLE — este es el plan FINAL de la fase y el candidato natural a cortar si la fase crece. Sin él, ESPACIO-03 ya está cumplido funcionalmente por el advisory lock (Plan 01). Si se recorta, documentarlo como hardening futuro (ya previsto en `<deferred>` de CONTEXT). El advisory lock SOLO ya da la garantía de concurrencia; este backstop es defensa en profundidad (Pitfall 6).

Purpose: El advisory lock es app-logic dentro del RPC; un error de lógica (ej. olvidar un espacio del set) no lo detecta nadie. El EXCLUDE de la proyección es declarativo: si dos reservas no se serializaran, el insert de proyección choca con 23P01 y aborta. Es barato (tabla + trigger + EXCLUDE) y es el único backstop cross-bucket (el 013 es por bucket, no por espacio).
Output: appointment_spaces + EXCLUDE + triggers, amendados al MISMO archivo `042_*.sql` (aún no aplicado a prod), schema.sql regenerado.
</objective>

  <action>
Agregar al `042_*.sql` (tras la tabla appointment_spaces) dos triggers en `appointments`, vía funciones plpgsql SECURITY DEFINER SET search_path = public:

1. `appointment_spaces_populate()` AFTER INSERT ON appointments FOR EACH ROW: si `NEW.status IN ('confirmed','pending_payment')`, `INSERT INTO appointment_spaces (appointment_id, business_id, space_id, slot) SELECT NEW.id, NEW.business_id, asp.space_id, tsrange(NEW.date + NEW.time, NEW.date + NEW.time + make_interval(mins => COALESCE(NEW.duration_minutes, 30))) FROM agenda_spaces asp WHERE asp.business_id = NEW.business_id AND asp.professional_id = NEW.professional_id`. Para una agenda SIN espacios la subconsulta devuelve 0 filas → no inserta nada (cero overhead/regresión para cupos/individual). La F11 expande a 3 filas (A,B,C), cada cruzada a 1 — cada espacio aparece una sola vez por appointment (Pitfall 3: no choca consigo misma). El EXCLUDE de Task 1 entonces choca SOLO contra filas de OTROS appointments con el mismo espacio y slot solapado.

2. `appointment_spaces_cleanup()` AFTER UPDATE OF status ON appointments FOR EACH ROW: si `NEW.status` deja de ocupar (`NEW.status NOT IN ('confirmed','pending_payment')`, ej. cancelled/expired) y `OLD.status` ocupaba, `DELETE FROM appointment_spaces WHERE appointment_id = NEW.id` (Pitfall 4: el espacio se libera al cancelar/expirar; espejo del WHERE de status de 013). El core ya pasa los holds vencidos a cancelled ANTES del RPC → el trigger los limpia.

Comentar en español: el INSERT de la proyección corre dentro de la MISMA transacción del insert de appointments (el RPC del Plan 01), así que el EXCLUDE es el respaldo — si el advisory lock fallara, el insert de proyección choca con 23P01 y aborta toda la transacción (incluido el insert del appointment). Documentar que el backstop aplica a reservas creadas TRAS la migración (sin backfill de turnos previos — RESEARCH §Runtime State Inventory).
  </action>
