<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| navegador (dueño) → endpoint | request autenticada del dueño; el body (incl. `notify`) es input a re-validar |
| endpoint → business_secrets | lectura service-role acotada por `business.id` (resend key/from) |
| endpoint → Resend | POST del mail al `clientEmail` del turno recién creado |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | mail al cliente | mitigate | `business` resuelto por owner_id; `secrets` vía `getBusinessSecrets(business.id)` (acotado al propio negocio); el mail va SOLO a `clientEmail` del turno recién creado y con `cancelToken` de ESE turno. Cero datos de otro tenant |
| T-05-02 | Spoofing/Tampering | flag `notify` en el body | accept | El actor ES el dueño autenticado (sesión, anon+RLS, tenant por owner_id). Forzar `notify=true` solo manda un mail al cliente que él mismo cargó — es la feature, no un abuso. Además gateado por `clientEmail` presente |
| T-05-03 | Denial of Service | envío de mail | accept + mitigate | Un mail por alta (acción humana del dueño); best-effort en `after()` con try/catch → si Resend falla, se loguea y el alta NO se rompe |
| T-05-04 | Tampering (header injection) | `From` = `businessName` | mitigate | Reusa `resolveSender`/`fromDisplayName` (Plan 01) que sanitiza el display name |
| T-05-06 | Tampering (bypass integridad) | alta manual | mitigate | El mail es puramente aditivo en `after()`: NO toca el core, ni `requireDeposit:false`, ni el anti-doble-booking, ni el after() de Google Calendar |
| T-05-SC | Tampering (supply chain) | npm installs | accept | Sin nuevas dependencias en la fase (Resend por `fetch` crudo, checkbox nativo) |
</threat_model>
