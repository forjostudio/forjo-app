<objective>
Backstop server anti-tampering (BOOK-WINDOW-03): `app/api/booking/create` rechaza una reserva pública con fecha fuera de la ventana aunque el cliente manipule la request. Es la capa de AUTORIDAD del enforcement en 3 capas (D-08: calendarios/UX + backstop/autoridad + disponibilidad/opcional) — el cap del calendario (Plan 03) es solo UX y se puede saltear. El backstop solo gatea reservas NUEVAS (D-06): no toca turnos ya reservados.

El route ya lee `businesses` con service-role (L52-54); basta sumar las 2 columnas al select y validar con el helper compartido `isDateOutOfWindow(business, date)` (Plan 01, hora AR) TEMPRANO: tras el gate de `plan_status` (L66-68), ANTES del insert de `client` (L86) — así una fecha fuera de ventana no deja `clients` huérfanos (Pitfall 3). Error `date_out_of_window` / 400 (validación de input, consistente con `missing_fields`).

El alta manual (`app/api/appointments/create`) es un archivo distinto que NO se toca → queda exenta por construcción (BOOK-WINDOW-03: el dueño carga con cualquier anticipación). Se agrega un test de no-regresión que lo demuestra vía el core compartido.

Purpose: es la garantía real de la fase (lo que verifica el secure-phase gate). Depende de Plan 01 (helper + tipos + columnas en la vista/tabla).
Output: backstop cableado + test de exención.
</objective>
